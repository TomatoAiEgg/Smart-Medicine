# 第一阶段编码落地方案

本文档记录第一阶段编码顺序。正式编码前以 `docs/03_架构设计/重构方案.md` 为上位依据。

## 1. 现状

当前项目已经完成第一轮文档整理和后端基础骨架：

- 旧系统分析已覆盖应用清单、业务全景、接口清单、数据库表说明和问题清单。
- 重构规格已覆盖第一版范围、新旧功能映射、新旧接口映射、新旧数据表映射和 AI 后续规划。
- 架构设计已覆盖重构方案、数据库设计、业务流程、RocketMQ 事件设计、PDA 与 MES 模拟方案；其中 `重构方案.md` 是当前唯一总方案入口。
- 后端已建立 Maven 多模块工程，包含 `zhyf-common`、`gateway`、`auth-institution`、`order-service`、`prescription-service`、`message-service`、`ops-service`。
- `order-service` 已具备机构推单、订单/处方基础落库、状态日志、Outbox 写入的最小链路。
- `message-service` 已按 `consumer`、`publisher`、`application`、`infrastructure` 拆包，具备 Outbox 发布和 `ORDER_CREATED` 消费骨架。
- 部署运维已记录老笔记本 Ubuntu 中间件主机、PostgreSQL、Redis、RocketMQ 和本地开发连接方式。

当前还未完成完整网关鉴权、机构配置服务、后台履约、PDA/MES、物流、回调、前端管理台。

## 2. 第一阶段目标

第一阶段目标不是一次性完成旧系统全量业务，而是先建立可持续开发的基础骨架，并跑通第一条最小业务链路：

```text
机构推单
 -> gateway 接入
 -> 机构签名/IP 校验
 -> order-service 创建订单
 -> order-service 临时创建处方和明细
 -> 写 event_outbox
 -> message-service 发布 RocketMQ 事件
 -> message-service 消费 ORDER_CREATED 并记录校验结果
 -> 校验通过后生成待审核任务
 -> 订单审核接口可推进审核状态
 -> 后台或接口可查询订单状态
```

第一阶段完成后，应具备：

- 后端多模块 Maven 工程能编译。
- 基础服务能启动。
- Flyway 能初始化第一批表。
- 机构推单接口有兼容旧系统的入口形态。
- 订单、处方、处方明细、状态日志、Outbox 能落库。
- RocketMQ 事件发送链路有最小可验证实现。
- 本地/云端开发配置不泄露敏感密码。

## 3. 变更范围

### 3.1 后端

在 `backend` 下创建 Maven 多模块工程：

```text
backend
├── pom.xml
├── zhyf-common
├── gateway
├── auth-institution
├── order-service
├── prescription-service
├── message-service
└── ops-service
```

本阶段暂不创建或暂不实现：

- `decoction-service`
- `logistics-service`
- `callback-service`

原因：第一阶段先跑通机构推单到事件发布链路，PDA/MES、物流、回调进入第二阶段。

### 3.2 数据库

在 `infra/flyway` 或后端资源目录中创建第一批 PostgreSQL migration：

- `tenant`
- `institution`
- `institution_app`
- `institution_ip_whitelist`
- `order_main`
- `order_status_log`
- `prescription`
- `prescription_detail`
- `event_outbox`
- `message_consume_log`
- `api_access_log`

字段以 `docs/03_架构设计/数据库设计.md` 为准，第一阶段只落必要字段。

### 3.3 接口

第一阶段实现最小接口：

| 接口 | 归属 | 说明 |
| --- | --- | --- |
| `POST /api/institution/createOrder` | `gateway` -> `order-service` | 兼容旧机构推单入口 |
| `GET /api/admin/orders/{orderNo}` | `order-service` | 查询订单基础信息 |
| `GET /actuator/health` | 各服务 | 健康检查 |

### 3.4 消息

第一阶段定义并实现：

- Topic：`zhyf-order-event`
- Tag：`ORDER_CREATED`
- Outbox 状态：`NEW`、`SENT`、`FAILED`
- 事件唯一 ID：`event_id`
- 消费幂等表：`message_consume_log`

### 3.5 前端

第一阶段前端只做最小管理入口，视后端进度决定是否落地：

- 若后端骨架和接口耗时较多，前端暂缓。
- 若后端主链路完成，可创建 `frontend/admin-web` Vue3 骨架和订单查询页。

## 4. 不变范围

第一阶段不做以下内容：

- 不接真实 PDA。
- 不接真实 MES。
- 不接真实顺丰/EMS。
- 不做社康真实拉单。
- 不做医院查单完整权限。
- 不做地址补录页面。
- 不做 AI 功能。
- 不清理旧系统数据库。
- 不修改旧项目源码。
- 不写入明文云服务器密码或数据库密码。

## 5. 技术约束

- Java 使用 21。
- Spring Boot 使用 3.x。
- PostgreSQL 作为唯一新库。
- Redis 只用于缓存、幂等、限流等非核心事实数据。
- RocketMQ 替代旧 ActiveMQ。
- 数据库结构变更必须有 migration。
- 外部接口请求和响应 DTO 与内部领域模型分离。
- 状态推进必须通过状态机或明确的应用服务方法，不允许随意直接赋值。
- 所有核心表带 `tenant_id` 或可追溯机构归属。

## 6. 风险

| 风险 | 影响 | 处理方式 |
| --- | --- | --- |
| 旧推单字段复杂 | 第一阶段 DTO 可能无法一次覆盖全部字段 | 先实现核心字段，保留 raw_payload 和扩展字段，后续按旧接口清单补齐 |
| 服务拆分过早导致联调复杂 | 多服务启动成本高 | 第一阶段只保留必要服务，内部调用可先 HTTP 或模块内清晰边界，后续再演进 |
| RocketMQ 联调受环境影响 | 事件发送验证失败 | 保留 Outbox 表作为事实来源，RocketMQ 失败可重试 |
| migration 字段和后续设计不一致 | 后续调整成本 | 第一批表只落稳定核心字段，避免过度设计 |
| 规范来自其他项目 | 存在手机脚本/APK 等无关条款 | 与本项目无关条款忽略，保留“先方案、再确认、再落地”的流程 |

## 7. 落地顺序

1. 创建后端 Maven 父工程和公共依赖管理。
2. 创建 `zhyf-common`，放置通用响应、异常、审计基类、状态枚举、签名工具。
3. 创建 `auth-institution`，实现机构应用、密钥、白名单基础模型和查询能力。
4. 创建 `order-service`，实现订单创建、订单查询、订单状态日志。
5. 保留 `prescription-service` 模块骨架，处方创建暂由 `order-service` 承接，后续迁移。
6. 创建 Flyway migration，初始化第一批表。
7. 创建 `message-service`，实现 Outbox 扫描和 RocketMQ 发布骨架。
8. 创建 `gateway`，实现 `/api/institution/createOrder` 路由、签名校验和访问日志。
9. 增加基础单元测试和最小集成测试。
10. 更新项目记录和联调记录。

## 8. 验证方式

第一阶段完成后至少验证：

- `mvn test` 通过。
- Flyway migration 可在空 PostgreSQL 库执行成功。
- 各服务健康检查正常。
- 使用测试机构密钥推单成功。
- 重复推单返回幂等结果或明确错误。
- 错误签名被拒绝。
- 错误 IP 被拒绝或记录为拒绝。
- 订单、处方、处方明细、状态日志、Outbox 均有落库记录。
- Outbox 事件能进入 `SENT` 或失败后保留可重试状态。
- `ORDER_CREATED` 消费后能写入校验记录；校验通过时生成 `ORDER_REVIEW` 待审核任务，校验拒绝时不生成任务。
- 审核过渡接口能查询待审核任务，并在审核通过/拒绝时推进订单和任务状态。

- 继续补齐 `gateway` 签名、白名单、访问日志和路由。
- 补齐 `auth-institution` 机构应用配置查询能力。
- 完成 `message-service` 消费幂等、`ORDER_CREATED` 校验记录和失败重试策略。
- 前端管理台只做可选最小订单查询页，不作为第一阶段阻塞项。
- PDA/MES、物流、回调、社康、医院查单、地址补录进入后续阶段。
