# 旧系统业务全景与重构功能规格说明书

本文档是智能药房 SaaS 平台重构的核心规格文档，用于把旧系统业务逻辑、新系统功能边界、模块拆分、兼容策略和第一版交付原则统一到同一份说明中。

详细旧系统分析见：

- `docs/01_旧系统分析/旧系统应用清单.md`
- `docs/01_旧系统分析/旧系统业务全景.md`
- `docs/01_旧系统分析/旧系统接口清单.md`
- `docs/01_旧系统分析/旧系统数据库表说明.md`

## 1. 重构目标

第一目标是完整替代旧系统核心业务能力，而不是先做新功能。

新系统需要完成以下目标：

1. 保留旧系统已承载的业务闭环。
2. 统一订单、处方、履约、物流、设备、回调、日志等领域模型。
3. 把旧系统分散在后台、API、PDA、异步任务、外围系统中的状态变更收敛为可审计状态机。
4. 保留对旧机构接口、PDA、物流、社康、医院查单、地址补录等关键外部能力的兼容。
5. 为后续 SaaS 化、多机构隔离、AI 提效和运营分析预留结构化数据基础。

不在第一阶段追求：

- 重做业务模式。
- 先上 AI 自动决策。
- 大幅改变医院/机构对接协议。
- 强行砍掉旧系统仍在使用的外部流程。

## 2. 旧系统业务边界

旧系统由多个应用共同组成：

| 旧应用 | 业务角色 | 新系统定位 |
| --- | --- | --- |
| `zhyfsupport` | 平台运营后台，处理审核、调剂、复核、打包、物流、设备、权限、统计 | 平台管理后台 |
| `zhyf-api` | 对外 API，承接机构推单、PDA、物流、设备、定制接口 | API 网关 + 机构/设备适配器 |
| `zhyf-async` | ActiveMQ 消费、订单校验、审方、回调、短信 | 异步任务/事件消费者 |
| `sc-zhyf-api` | 社康/智慧云/医院定制 API 变体 | 机构适配器 |
| `sc-zhyf-communityhospital` | 社康定时拉单、上传、状态回写 | 社康集成服务 |
| `hospital-query` | 医院侧订单/处方/物流查询 | 机构门户 |
| `sc-address-input` | 患者地址补录和医院回推 | 地址补录门户 + 地址适配器 |

## 3. 新系统领域模块

建议第一版按以下领域模块拆分。

| 模块 | 核心职责 |
| --- | --- |
| 租户与机构模块 | 机构、煎煮中心、接口密钥、白名单、机构配置 |
| 用户权限模块 | 平台用户、机构用户、角色权限、数据范围 |
| 订单模块 | 订单创建、查询、取消、地址修改、订单状态机 |
| 处方模块 | 处方、处方明细、审核、处方修改、处方状态 |
| 审核调剂复核模块 | 后台审核、调剂打单、复核、水桶绑定、人工修正 |
| 煎煮作业模块 | PDA 绑定设备、加水、开始煎煮、完成煎煮、取消/终止 |
| 设备模块 | 水桶、煎煮机、PDA、打码机、设备占用、设备关系 |
| 物流模块 | 物流下单、面单、打包、自提、发货、轨迹回调、合并发货 |
| 回调与消息模块 | 机构状态回调、消息发送、重试、消费记录、Outbox |
| 社康集成模块 | 社康消息拉取、订单转换、上传、状态回写、失败重试 |
| 机构门户模块 | 医院查单、订单详情、物流轨迹、导出 |
| 地址补录模块 | 地址补录页面、地址保存、医院差异化回推、日志 |
| 操作日志与审计模块 | 后台操作、PDA 操作、状态流转、人工干预记录 |
| 统计报表模块 | 订单、处方、人员绩效、设备、物流、机构统计 |

## 4. 主业务流程

### 4.1 标准订单履约流程

```text
机构推单
 -> 验签、白名单、幂等校验
 -> 创建订单、处方、处方明细
 -> 异步药品匹配、订单校验、处方校验、审方信息回写
 -> 后台审核
 -> 调剂打单
 -> 复核
 -> 代煎处方绑定水桶/煎煮机
 -> PDA/设备开始煎煮
 -> PDA/设备完成煎煮
 -> 打包/面单/自提
 -> 物流发货/运输/签收
 -> 回调机构订单状态
 -> 医院/机构查询订单
```

### 4.2 核心订单状态

第一版应先兼容旧系统状态码，再在内部建立结构化状态机。

| 旧状态 | 含义 | 新系统阶段 |
| --- | --- | --- |
| `1` | 已创建 | 创建 |
| `5` | 审核中/已接单 | 待审核 |
| `8` | 用户待确认 | 待确认 |
| `15` | 调剂中 | 调剂 |
| `20` | 复核完成 | 复核完成 |
| `23` | 已浸泡 | 浸泡 |
| `25` | 煎煮中 | 煎煮中 |
| `27` | 煎煮完成 | 煎煮完成 |
| `28` | 包装完成 | 包装过程 |
| `29` | 质检完成 | 质检 |
| `30` | 打包完成 | 打包完成 |
| `35` | 物流分拨 | 物流分拨 |
| `36` | 已发货 | 已发货 |
| `37` | 运输中 | 运输中 |
| `40` | 签收 | 已完成 |
| `97` | 用户拒收 | 终态异常 |
| `98` | 审核失败 | 终态异常 |
| `99` | 用户取消 | 终态取消 |

第一版必须保留对外状态码兼容。

内部建议增加：

- 订单主状态。
- 处方作业状态。
- 煎煮任务状态。
- 物流状态。
- 回调状态。
- 人工干预状态。

避免继续只依赖 `order_status_follow` 字符串。

## 5. 关键业务规则

### 5.1 机构推单

必须保留：

- Header + Body 签名校验。
- IP 白名单校验。
- 按机构和医院处方号做重复拦截。
- Redis 或等效机制短时防重复提交。
- 多处方落同一订单。
- 订单、处方、处方明细事务落库。
- 创建订单后异步处理校验、审方、药品匹配。

新系统补强：

- 使用数据库唯一约束和幂等记录替代单纯 Redis 防重。
- 创建订单成功后通过 Outbox 发事件。
- 外部请求和响应入库，便于联调追踪。

### 5.2 后台审核

必须保留：

- 只有 `order_status <= 5` 的订单可审核。
- 审核通过后订单状态为 `15`。
- 写处方审核人、审核时间、审核结果。
- 更新处方明细。
- 写操作日志和履约轨迹。
- 发送订单状态回调。
- 需要支持订单锁定/解锁。

新系统补强：

- 审核修改前后应形成版本记录。
- 审核操作应独立成 `audit_task` 或 `prescription_audit_record`。

### 5.3 调剂与复核

必须保留：

- 调剂员工号校验。
- 调剂后写调剂人、调剂时间、打印状态。
- 复核员工号校验。
- 复核前校验处方重量。
- 代煎处方复核时校验加水量和水桶号。
- 复核时写水桶绑定记录。
- 所有处方调剂和复核完成后订单状态到 `20`。
- 发送 `20` 状态回调。

新系统补强：

- 避免旧系统中补调剂和复核都可能发送 `20` MQ 的重复回调问题。
- 调剂、复核、打印应形成独立作业记录。

### 5.4 PDA 与煎煮

必须保留：

- PDA 登录。
- 查询可用煎煮机。
- 处方绑定煎煮机。
- 设备占用和释放。
- 开始煎煮：订单到 `25`，处方 `boil_status=1`。
- 完成煎煮：订单到 `27`，处方 `boil_status=2`，释放设备。
- 取消/终止煎煮。
- PDA 操作日志。
- 履约轨迹。

新系统补强：

- 建立 `decoction_task` 和 `device_occupancy`。
- 设备状态变更要防并发抢占。
- PDA 上报需要幂等。

### 5.5 物流、打包与自提

必须保留：

- 面单前校验订单状态至少为 `20`。
- 创建物流单。
- 打包完成状态 `30`。
- 写物流公司、运单号、重量、包裹数、打包时间。
- 发送发货短信。
- 发送 `30` 状态回调。
- 顺丰/EMS 回调更新 `36/37/40`。
- 自提直接完成到 `40`。
- 合并发货。

新系统补强：

- 物流单和订单分离建模。
- 合并发货用 `shipment_group` 表达。
- 物流轨迹供应商状态和平台状态分开。

### 5.6 社康集成

必须保留：

- 按深圳区域定时拉取社康消息。
- 本地保存消息防漏单。
- 拉取社康订单详情。
- 转换成智慧药房订单请求。
- 上传成功后回写社康状态。
- 回写失败重试。
- 退费状态回写。

新系统补强：

- 区域、机构、接口地址、拉取频率配置化。
- 使用统一集成任务和重试表。
- 社康订单转换要记录源数据和目标数据。

### 5.7 医院查单

必须保留：

- 医院用户登录和权限。
- `companyNums` 数据范围。
- 订单/处方列表。
- 订单详情。
- 物流轨迹。
- 导出。

新系统补强：

- 机构门户与平台后台分开。
- 查询结果脱敏策略明确。

### 5.8 地址补录

必须保留：

- 按机构跳转地址补录页面。
- 保存历史地址。
- 按医院差异化回推地址。
- 推送日志。
- Redis 防重复提交。
- 微信 JSSDK 签名能力。

新系统补强：

- 地址保存与医院回推解耦。
- 医院协议适配器化。
- 地址回推失败支持重试和人工补推。

## 6. 兼容策略

### 6.1 对外接口兼容

第一版应优先兼容旧接口协议，尤其是：

- 机构推单 `/createOrder`
- 订单取消 `/cancelOrder`
- 订单查询 `/queryOrder`
- 地址修改 `/updateOrderAddress`
- PDA 接口
- 物流回调
- 智慧云订单同步
- 龙岗中心医院 XML/RSA 接口
- 社康回调地址

新接口可以内部重命名，但外部适配层必须维持旧协议。

### 6.2 状态码兼容

外部机构仍使用旧状态码。

新系统内部可以增加结构化状态，但对外回调、查询、旧接口响应必须兼容旧状态。

### 6.3 数据迁移兼容

旧数据迁移时必须保留：

- 订单号。
- 处方号。
- 医院处方号。
- 机构编号。
- 订单状态。
- 物流单号。
- 操作日志。
- 履约轨迹。
- 回调记录。

可废弃但需归档：

- 重复冗余字段。
- 字符串状态流转字段。
- 无实际使用的临时调试字段。

## 7. 第一版交付原则

第一版以“替代旧系统核心生产流程”为目标。

必须做到：

- 能接单。
- 能审核。
- 能调剂。
- 能复核。
- 能煎煮。
- 能打包。
- 能发货。
- 能回调。
- 能查单。
- 能补地址。
- 能处理社康。
- 能看日志和追异常。

暂不做：

- AI 自动审核。
- 大屏驾驶舱。
- 复杂 BI。
- 自动调度设备最优算法。
- 非必要 UI 重设计。
- 新商业模式功能。

## 8. 验收口径

第一版重构验收应按业务链路验收，不只按接口是否返回成功验收。

核心验收链路：

1. 机构推单到后台审核。
2. 审核到调剂。
3. 调剂到复核。
4. 复核到 PDA 煎煮。
5. 煎煮到打包。
6. 打包到物流回调签收。
7. 每个状态回调机构。
8. 医院查单能查到完整状态和轨迹。
9. 社康拉单能完成上传和回写。
10. 地址补录能保存并回推医院。
11. 人工初始化、取消、签收、走流程有审计。

## 9. 后续增强方向

核心重构完成后，再考虑：

- AI 提效，见 `docs/02_重构规格/AI提效需求规划.md`。
- 运营报表增强。
- 设备监控增强。
- 自动异常预警。
- 机构自助配置。
- 更细的数据分析和绩效管理。
