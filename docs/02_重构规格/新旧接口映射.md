# 新旧接口映射

本文档记录旧接口到新接口的兼容、替代、废弃和拆分策略。

## 1. 映射策略

第一版采用“外部兼容、内部重构”的策略。

- 外部机构、PDA、物流、社康、医院地址补录接口优先兼容旧协议。
- 新系统内部可以使用更清晰的 REST/RPC/Event 接口。
- 对医院定制协议使用适配器，不直接污染核心订单服务。
- 对异步消息使用事件模型，不继续依赖业务代码直接发送 MQ。

## 12. 异步队列映射

旧系统 ActiveMQ 队列建议映射为 RocketMQ Topic 或内部事件。

| 旧队列 | 新事件/Topic | 消费职责 | 策略 |
| --- | --- | --- | --- |
| `createOrderSuccess.service.queue` | `OrderCreated` | 药品匹配、校验、审方、位置校验 | 替代 |
| `reviewFinishPushRecipe.service.queue` | `PrescriptionReviewed` | 推送处方到煎煮/设备服务 | 替代 |
| `sendMsgMessageMQ.service.queue` | `NotificationRequested` | 短信/通知发送 | 替代 |
| `orderInfoValidationMQ.service.queue` | `OrderValidationRequested` | 修改后重新校验 | 替代 |
| `threeAudit.service.queue` | `InstitutionAuditCallbackRequested` | 旧审核回调兼容 | 评估 |
| `orderStatusChange.service.queue` | `OrderStatusChanged` | 回调机构订单状态 | 替代 |

消息要求：

- 业务库写入和事件写入同事务。
- 消费侧有幂等记录。
- 回调有重试和失败记录。
- 不再只依赖 `zhyf_mq_message`。

## 13. 废弃或弱化接口

| 旧接口/能力 | 策略 | 原因 |
| --- | --- | --- |
| 测试 Controller | 废弃 | 非生产能力 |
| 旧调试输出接口 | 废弃 | 安全风险 |
| 旧直接暴露内部 ID 获取接口 | 评估 | 可由内部服务替代 |
| 重复的第三方审核回调队列 | 评估合并 | 与订单状态回调重叠 |

## 14. 当前结论

外部接口第一版以兼容为主。

内部接口第一版可以按领域重新设计，但必须通过适配层保证旧机构、PDA、物流、社康、地址补录继续可用。
