# 压测方案

## 1. 压测目标

压测目标不是单接口跑分，而是验证真实业务链路在高并发下的稳定性。

重点验证：

- 机构批量推单能力。
- PDA 并发扫码能力。
- MES 状态上报能力。
- RocketMQ 消费能力。
- PostgreSQL 查询和写入性能。
- 回调失败重试能力。

## 2. 工具

- JMeter：适合构造复杂 HTTP 场景。
- k6：适合脚本化、持续压测和 CI 集成。
- RocketMQ Dashboard：观察消息堆积。
- Prometheus + Grafana：观察接口、数据库、JVM、MQ 消费指标。

## 3. 压测场景

### 场景一：机构批量推单

流程：

```text
机构推单
 -> 创建订单
 -> 创建处方
 -> 创建处方明细
 -> 写 Outbox
 -> 发布 ORDER_CREATED / PRESCRIPTION_CREATED
```

参数：

- 每单 1 到 3 张处方。
- 每张处方 10 到 25 味药。
- 100、300、500、1000 并发梯度。

指标：

- P95 响应时间。
- 错误率。
- 订单创建 TPS。
- Outbox 堆积量。

### 场景二：PDA 并发扫码绑定

流程：

```text
PDA 登录
 -> 扫码绑定处方
 -> 写绑定记录
 -> 状态机推进
 -> 发布 PDA_PRESCRIPTION_BOUND
```

异常模拟：

- 同一处方重复扫码。
- 同一处方多个 PDA 同时扫码。
- 客户端超时后重复提交。

验收：

- 同一处方只能绑定成功一次。
- 重复请求返回幂等结果。
- 状态日志不重复。

### 场景三：MES 状态上报

流程：

```text
MES 上报开始煎煮
 -> MES 上报完成煎煮
 -> 订单/处方/煎煮状态推进
 -> 发布 DECOCTION_STARTED / DECOCTION_FINISHED
```

异常模拟：

- 重复上报。
- 乱序上报。
- 并发上报。

### 场景四：物流轨迹回调

流程：

```text
物流轨迹批量回调
 -> 写 logistics_trace 分区表
 -> 更新物流状态
 -> 签收后发布 LOGISTICS_SIGNED
```

指标：

- 写入 TPS。
- 分区表查询耗时。
- 签收事件延迟。

### 场景五：机构回调补偿

流程：

```text
业务事件触发回调
 -> 模拟机构接口失败
 -> callback-service 重试
 -> 超过次数进入死信
 -> 人工重放
```

指标：

- 回调成功率。
- 平均重试次数。
- 死信数量。
- 重放成功率。

### 场景六：后台分页查询

查询对象：

- 订单列表。
- 处方列表。
- 物流轨迹。
- 回调记录。
- 操作日志。

要求：

- 禁止深分页全表扫。
- 必须按 tenant_id、时间、状态建立查询条件。
- 大表必须命中分区和索引。

## 4. 性能目标

初始目标：

- 推单接口 P95 小于 500ms。
- PDA 操作接口 P95 小于 300ms。
- MES 上报接口 P95 小于 300ms。
- 回调接收接口 P95 小于 300ms。
- 后台常用查询 P95 小于 800ms。
- RocketMQ 消息无长时间堆积。

## 5. 压测报告内容

每次压测输出：

- 压测时间。
- Git 提交版本。
- 环境配置。
- 并发数。
- 总请求数。
- 平均响应时间。
- P95 / P99。
- 错误率。
- 数据库 CPU / 慢 SQL。
- RocketMQ 堆积。
- JVM GC 情况。
- 优化结论。
