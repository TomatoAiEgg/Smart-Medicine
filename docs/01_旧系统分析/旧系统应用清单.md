### 2.1 `zhyfsupport`

旧后台支持系统，定位为智慧药房煎药配送业务的后台管理端。

已确认信息：

| 路径 | `D:\workproject\chengyou\1\zhyfsupport` |

| 主要入口 | `src/main/java/com/zhyf/support/controller` |
| 页面形态 | JSP 后台页面 |
| 核心配置 | `src/main/resources/spring`、`src/main/resources/mybatis`、`jdbc.properties`、`mq.xml` |

主要业务模块：

| 模块 | 代表入口 | 业务含义 |

| 短信 | `SmsController` | 短信发送与模板 |

重构判断：

- 该应用是旧系统业务后台的主要来源，应作为后台管理、人工补偿、配置管理的业务依据。

- Controller、Service、Mapper 要继续深挖，用于补充旧后台功能清单和新 `admin-web` 页面范围。

### 2.2 `zhyf-api`

旧对外接口系统，定位为医疗机构、PDA、物流、智能云平台等外部系统的 API 入口。

已确认信息：

| 项 | 内容 |
| --- | --- |
| 路径 | `D:\workproject\chengyou\1\zhyf-api` |
| 模块 | `zhyf-common`、`zhyf-gateway`、`zhyf-async` |
| 技术栈 | Spring Boot、MyBatis、MySQL、ActiveMQ/消息组件、HTTP 第三方接口 |
| 主要入口 | `zhyf-gateway/src/main/java/com/zhyf/api/controller` |

主要 Controller：

| Controller | 代表接口 | 业务含义 |
| --- | --- | --- |
| `OrderApiController` | `/createOrder`、`/cancelOrder`、`/queryOrder`、`/updateOrderAddress`、`/getOrderId` | 机构推单、取消订单、订单查询、地址更新、订单号获取 |
| `PDAController` | `/pdaUserLogin`、`/pdaGetDecoctEquip`、`/pdaBindPrescription`、`/pdaOpDecoctStatus`、`/pdaGetCanOpRecipeList`、`/pdaLabelPrint`、`/pdaLogisticsOutbound` | PDA 登录、设备查询、处方绑定、煎煮状态操作、标签打印、物流出库 |
| `LogisticsController` | `/logistics/createLogisticsOrder`、`/logistics/cancelLogisticsOrder`、`/logistics/queryBillPrintInfo`、`/logistics/notify` | 创建物流单、取消物流、面单查询、顺丰路由回调 |
| `MachineApiController` | 待深挖 | 煎煮机或设备侧接口 |
| `SmartCloudOrderApiController` | 待深挖 | 智能云订单接口 |
| `SzlgzxyyOrderApiController` | 待深挖 | 深圳龙岗中心医院相关接口 |

初步主链路：

```text
机构调用 /createOrder
 -> OrderService.createOrder
 -> 写订单、处方、处方明细、消息记录
 -> PDA 调用绑定和煎煮接口
 -> PDA 或后台触发物流出库
 -> 物流回调 /logistics/notify 更新订单状态
 -> MQ 推送订单状态消息给外部系统
```

重构判断：

- 该应用是新系统 `gateway`、`order-service`、`prescription-service`、`decoction-service`、`logistics-service`、`callback-service` 的主要来源。
- 外部接口不应简单照搬 URL，需要判断哪些必须兼容、哪些可以在新系统中重命名。
- `/createOrder`、PDA、物流回调应作为后续深挖优先级最高的接口。

已确认信息：

| 项 | 内容 |
| --- | --- |
| 路径 | `D:\workproject\chengyou\sc-zhyf-communityhospital` |
| 打包方式 | JAR |
| 主启动类 | `com.zhyf.community.MyApplication` |
| 技术栈 | Spring Boot、MyBatis、MySQL、Thymeleaf、Ehcache、HTTP/XML |
| 访问路径 | `/szc` |
| 主要 Controller | `IndexController`、`OffOperController`、`RefundController` |
| 主要任务 | `AreaOrderTask`、`LocalMessageTask`、`MultiThreadScheduler` |

核心业务：

```text
定时扫描各区社康消息
 -> 判断消息类型是否为上传订单
 -> 根据机构、病人、门诊号获取订单信息
 -> 保存本地消息记录防重复
 -> 上传订单到智慧云煎药中心
 -> 失败时写日志并短信通知
```

- 罗湖
- 南山

- 该应用应归入新系统的 `callback-service` / `order-service` 外部接入适配能力，也可能拆为“社康适配器”。
- 其本地消息表和离线操作表代表旧系统的补偿能力雏形，需要在新系统中统一到 Outbox、重试和人工补偿。

## 3. 应用间关系初稿

```text
医院/社康/机构
   │
   ├── 直接推单到 zhyf-api / sc-zhyf-api
   │      └── 写订单、处方、处方明细、MQ 消息
   │
   ├── 社康消息由 sc-zhyf-communityhospital 定时拉取
   │      └── 获取社康订单后上传到智慧云煎药中心接口
   │
   ├── 患者地址补录由 sc-address-input 处理
   │      └── 按机构配置推送地址到医院接口
   │
   └── 医院查询由 hospital-query 提供
          └── 查询订单、处方、物流轨迹

药房/煎煮中心工作人员
   │
   ├── 使用 zhyfsupport 后台处理订单、处方、煎煮、物流、配置
   └── 使用 PDA 调 zhyf-api 的 PDA 接口完成绑定、煎煮和出库
```

## 4. 重构应用归属初稿

| 旧应用/能力 | 新系统归属 | 说明 |
| --- | --- | --- |
| `zhyf-api` 机构推单 | `gateway` + `order-service` + `prescription-service` | 对外入口、签名、订单处方创建 |
| `zhyf-api` PDA 接口 | `decoction-service` | PDA 登录、设备、绑定、煎煮状态、出库 |
| `zhyf-api` 物流接口 | `logistics-service` + `callback-service` | 创建物流单、物流回调、状态更新 |
| `zhyfsupport` 后台管理 | `admin-web` + 各业务服务 | 页面重做，业务能力保留 |
| `sc-zhyf-communityhospital` 社康拉单 | `callback-service` / 外部适配器 | 社康消息拉取、订单上传、失败补偿 |
| `hospital-query` 医院查单 | `admin-web` / 机构查询门户 | 数据权限和查询视图保留 |
| `sc-address-input` 地址录入 | `order-service` / 地址采集适配器 | 地址采集、推送日志、机构加密适配 |
