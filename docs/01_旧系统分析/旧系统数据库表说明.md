# 旧系统数据库表说明

本文档用于记录旧数据库表的业务含义、核心字段、关联关系、数据规模、历史问题和新系统领域归属。

## 1. 核心订单履约表

### 1.1 `zhyf_order`

订单主表。

| 字段 | 业务含义 |
| --- | --- |
| `order_id` | 平台订单号 |
| `company_num` | 机构编码，旧系统中通常对应接口 `appKey` |
| `storage_type` | 煎煮中心/仓储类型，来自机构配置 |
| `order_status` | 订单状态数字 |
| `order_status_follow` | 订单状态历史字符串，如 `5->25->27` |
| `sp_orderid` | 外部/兼容订单号；创建订单时默认等于平台订单号 |
| `callback_url` | 第三方机构状态回调地址 |
| `province`、`city`、`zone`、`addr_detail` | 收货地址 |
| `consignee`、`tel` | 收件人和电话 |
| `logistics_company_name`、`logistics_no` | 物流公司和运单号 |
| `package_time`、`outbound_time`、`sign_time` | 打包、出库、签收时间 |
| `version` | 版本号；旧系统仅部分更新时自增 |

已确认状态：

| 状态值 | 枚举 | 含义 |
| --- | --- | --- |
| `1` | `CREATEORDER` | 已创建 |
| `5` | `AUDITING` / `HAVEORDER` | 审核中/已接单 |
| `8` | `USER_CONFIRM` | 待确认 |
| `15` | `DISPENSING` | 调剂中 |
| `20` | `PREPARE_MED_FINISHED` | 复核完成 |
| `23` | `WATERSUCCESS` | 已浸泡 |
| `25` | `BOILING` | 煎煮中 |
| `27` | `BOILEND` | 煎煮完成 |
| `28` | `PACKAGEEND` | 包装完成 |
| `29` | `QUALITYCHECKEND` | 质检完成 |
| `30` | `PACKAGE_FINISHED` | 打包完成 |
| `35` | `LOGISTICS_DISPATCH` | 物流分拨 |
| `36` | `LOGISTICS_DELIVERY` | 已发货 |
| `37` | `LOGISTICS_TRANSIT` | 运输中 |
| `40` | `SIGNED` | 派送完毕/签收 |
| `97` | `USER_REFUSE` | 用户拒收 |
| `98` | `AUDIT_FAIL` | 审核失败 |
| `99` | `USER_CANCEL` | 用户取消 |

新系统归属：`order_main`、`order_status_log`。

### 1.2 `zhyf_prescription`

处方主表。

| 字段 | 业务含义 |
| --- | --- |
| `prescri_id` | 平台处方号，创建订单时使用 `orderId-序号` |
| `order_id` | 所属订单号 |
| `hos_prescri_num` | 医院处方号 |
| `company_num` | 机构编码 |
| `prescri_type` | 处方类型；PDA 煎煮仅允许代煎处方，代码里判断 `2` |
| `patient_name`、`patient_age`、`patient_gender` | 患者信息 |
| `doctor_name`、`prescri_diagnose` | 医生和诊断 |
| `amount`、`decoct_amount`、`per_pack_num`、`per_pack_dose` | 剂数、煎煮量、每剂包数、每包剂量 |
| `audit_*` | 审方相关字段 |
| `dispense_*` | 调剂相关字段 |
| `recheck_*` | 复核相关字段 |
| `soak_*` | 浸泡相关字段 |
| `boil_status` | 煎煮状态：0 未煎煮、1 进行中、2 完成、8 取消、9 终止 |
| `boil_time_start`、`boil_time_end`、`boil_user_name` | 煎煮时间和人员 |
| `out_med_*` | 出药相关字段 |
| `pack_*` | 包装相关字段 |
| `order_handle_floor` | 建议处理楼层 |
| `jyj_decoction_plan` | 煎煮机方案 |

新系统归属：`prescription`、`decoction_task`。

### 1.3 `zhyf_prescription_details`

处方明细表。

| 字段 | 业务含义 |
| --- | --- |
| `prescription_details_id` | 明细 ID，创建订单时使用 `prescriId-序号` |
| `prescri_id` | 所属处方 |
| `goods_name`、`goods_num` | 药品名称和医院/外部药品编码 |
| `dc_goods_num`、`dc_goods_name` | 药房/煎煮中心匹配后的药品编码和名称 |
| `dose`、`unit` | 剂量和单位 |
| `special_usage` | 特殊用法 |
| `status` | 明细状态 |
| `med_sale_price`、`med_settle_price` | 销售价和结算价 |
| `batch_no` | 批号 |

新系统归属：`prescription_detail`。

## 3. 物流与消息表

### 3.1 `zhyf_logis_info`

物流轨迹/履约轨迹表。

用途：

- 创建订单后写“煎药中心接单”。
- PDA 开始/完成煎煮写履约轨迹。
- 顺丰/EMS 回调写物流轨迹。

新系统归属：`logistics_trace` 或独立 `fulfillment_trace`。

### 3.2 `zhyf_mq_message`

MQ 结果记录表。

| 字段 | 业务含义 |
| --- | --- |
| `message_type` | 消息类型 |
| `order_id` | 订单号 |
| `prescri_id` | 处方号 |
| `result_code` | 结果；旧系统中 `0` 通常成功，`1` 失败 |
| `result_msg` | 返回或失败信息 |
| `resend_num` | 重发次数 |
| `resend_time` | 重发时间 |
| `remark` | 说明 |
| `storagetype` | 煎煮中心/仓储类型 |

已确认用途：

- 订单状态变化回调第三方后写记录。
- 旧第三方审核回调后更新记录。

新系统归属：`callback_record`、`message_consume_log`，部分能力应升级为 `event_outbox`。

## 4. 后台履约补充字段

### 4.1 `zhyf_order` 后台履约字段

第三轮确认的字段用途：

| 字段 | 业务含义 |
| --- | --- |
| `order_status` | 当前订单状态；后台审核、打包、自提、签收、人工流程均直接更新 |
| `order_status_follow` | 状态流转字符串，旧系统用 `->状态码` 追加 |
| `logistics_money` | 物流费；审核或面单前按规则计算 |
| `logistics_company_num`、`logistics_company_name` | 物流公司编号和名称 |
| `logistics_no` | 运单号；自提时旧系统写订单号 |
| `order_pkg_weight` | 包裹重量 |
| `order_pkg_num` | 包裹数或寄存编号 |
| `package_time` | 打包时间 |
| `outbound_time` | 出库时间 |
| `sign_time` | 签收时间 |
| `logistics_pay_method` | 物流付款方式 |
| `logistics_clerk_id` | 物流/出库操作人 |
| `order_remark` | 后台备注；订单走流程会追加备注 |

新系统建议：

- 将订单主状态、物流状态、包裹信息拆分，不要继续把所有履约信息堆在订单主表。
- `order_status_follow` 应替换为结构化 `order_status_log`。

### 4.2 `zhyf_prescription` 后台履约字段

第三轮确认的字段用途：

| 字段 | 业务含义 |
| --- | --- |
| `audit_user_id` | 审核人 |
| `audit_time` | 审核时间 |
| `audit_result` | 审核结果；取消处方时写 `1` |
| `dispense_user_id` | 调剂员工号 |
| `dispense_time` | 调剂时间 |
| `print_status` | 调剂/打印状态；`0` 未打印，`1` 已打印 |
| `recheck_user_id` | 复核员工号 |
| `recheck_time` | 复核时间 |
| `recheck_flow_pic_url` | 复核流程图片 |
| `water` | 加水量，代煎复核时必填 |
| `weight` | 处方重量，复核时必填 |
| `order_handle_floor` | 楼层建议 |
| `jyj_decoction_plan` | 煎药机方案 |
| `jyj_decoction_advice` | 煎药机建议描述 |
| `soak_time_start`、`boil_time_start` | 人工走流程或煎煮流程写入 |

新系统建议：

- 审核、调剂、复核应拆成独立作业记录，处方表只保留当前摘要字段。
- 复核图片应纳入附件表，记录业务类型、上传人和存储地址。

### 4.3 `zhyf_prescription_details`

处方明细表在后台审核保存时会被更新。

第三轮确认用途：

- 审核时保存处方明细修改结果。
- 异步校验保留按药品特殊用法、附子剂量推导煎药方案的逻辑。

新系统建议：

- 明细修改需要保留审计版本，尤其是审核阶段改药品剂量、用法、价格等信息。

## 8. 医院查单数据口径

来源应用：`hospital-query`

医院查单系统没有独立业务主表，主要复用以下旧系统核心表：

- `zhyf_order`
- `zhyf_prescription`
- `zhyf_prescription_details`
- `zhyf_logis_info`
- `zhyf_institutions`
- `zhyf_dict`

查询特点：

- 列表以 `zhyf_prescription` 为主表。
- 关联 `zhyf_order` 获取订单状态、收货地址、物流信息。
- 关联 `zhyf_institutions` 获取机构名称。
- 关联 `zhyf_dict` 获取处方类型和订单状态名称。
- 详情查询会关联物流轨迹和处方明细。

新系统归属：机构门户查询视图，可用只读投影表或查询模型实现。
