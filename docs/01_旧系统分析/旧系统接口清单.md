# 旧系统接口清单

本文档用于记录旧系统对外接口、后台接口、内部调用入口、请求参数、响应结构、落库行为和新系统兼容策略。

## 1. 说明

本文档先记录第二轮已从源码确认的主链路接口。后续会继续补充后台页面接口、社康定制接口、智能云接口和设备/MES 接口。

## 2. 机构订单接口

来源应用：`1\zhyf-api\zhyf-gateway`

Controller：`com.zhyf.api.controller.OrderApiController`

| 旧接口 | 方法 | 入参核心对象 | 主要业务 | 主要落库/更新 | 新系统归属 |
| --- | --- | --- | --- | --- | --- |
| `/healthCheck` | `@RequestMapping` | 无 | 健康检查 | 无 | `gateway` |
| `/createOrder` | `@RequestMapping` | JSON -> `CreateOrderParam` | 机构推单；创建订单、处方、处方明细 | `zhyf_order`、`zhyf_prescription`、`zhyf_prescription_details`、`zhyf_logis_info` | `gateway` + `order-service` + `prescription-service` |
| `/cancelOrder` | `@RequestMapping` | `CancelOrderParam` | 机构取消订单 | `zhyf_order`、日志表；具体状态需继续补充 | `order-service` |
| `/queryOrder` | `@RequestMapping` | `QueryOrderParam` | 查询订单状态 | 读 `zhyf_order`、处方 | `order-service` |
| `/updateOrderAddress` | `@RequestMapping` | `ApiRequest<UpdateAddrDTO>` | 更新订单收件地址 | `zhyf_order`、`zhyf_log_info` | `order-service` |
| `/getOrderId` | `@RequestMapping` | `ApiRequestHeader` | 获取平台订单号 | 只生成订单号，不落订单 | `order-service` |

### 2.1 `/createOrder` 已确认校验

| 校验 | 说明 |
| --- | --- |
| Header 校验 | `apiPermissionService.headerParamValidation` |
| 签名校验 | 使用 `orderTime`、`tel`、处方号列表参与 `createOrder` 签名 |
| 业务参数校验 | `validateCreateOrderData` |
| IP 白名单 | 按 `appKey` 和请求 IP 校验 |
| 防重复 | Redis `setNX(createOrder-{appKey}-{hosRecipeNos}, 1, 5秒)` |
| 重复处方 | 按 `companyNum + hos_prescri_num` 查未取消/拒收的旧处方 |

### 2.2 `/createOrder` 已确认响应

成功时返回：

| 字段 | 含义 |
| --- | --- |
| `orderId` | 平台订单号 |
| `recipeInfo[].recipeId` | 平台处方号，格式为 `orderId-序号` |
| `recipeInfo[].hosRecipeNo` | 医院处方号 |

### 3.1 PDA 煎煮状态

| `operStatus` | 旧系统含义 | 主要动作 |
| --- | --- | --- |
| `0` | 已绑定/待煎煮 | 绑定设备时写入绑定表 |
| `1` | 开始煎煮 | 更新绑定状态；写处方开始时间和 `boil_status=1`；订单状态改为 `25`；写轨迹；写煎煮机工作参数 |
| `2` | 煎煮完成 | 更新绑定状态；写处方结束时间和 `boil_status=2`；订单状态改为 `27`；设备释放；发订单状态 MQ；写轨迹 |
| `8` | 取消煎煮 | 写 `boil_status=8`；释放设备 |
| `9` | 终止煎煮 | 写 `boil_status=9`；释放设备 |

## 5. ActiveMQ 队列

来源应用：

- 生产者：`1\zhyf-api\zhyf-gateway`
- 消费者：`1\zhyf-api\zhyf-async`

队列常量：`com.zhyf.api.common.consts.MQConst`

| 队列 | 生产入口 | 消费入口 | 业务含义 |
| --- | --- | --- | --- |
| `createOrderSuccess.service.queue` | `MQProducer.sendCreateOrderSuccess` | `ActiveMqConsumer.createOrderSuccessNotify` | 创建订单成功后异步处理药品匹配、处方校验、楼层/设备建议、位置校验、审方平台调用 |
| `reviewFinishPushRecipe.service.queue` | `MQProducer.reviewFinishPushRecipe` | `ActiveMqConsumer.reviewFinishPushRecipe` | 复核完成后推送处方到煎煮系统 |
| `sendMsgMessageMQ.service.queue` | `MQProducer.sendMsg` | `SendMsgListener.sendMsg` | 异步短信/消息发送 |
| `orderInfoValidationMQ.service.queue` | 后台修改订单信息后由后台 MQ 推送服务触发 | `OrderValidationListener.orderValidation` | 后台修改后重新校验订单/处方 |
| `threeAudit.service.queue` | 后台审核通过时曾设计由 `sendThreeAuditMapMessage(orderId)` 触发；当前主版本发送逻辑已注释，仅写 `zhyf_mq_message` 记录 | `ThreeAuditListener.sendMsg` | 旧第三方审核回调：根据订单 `callbackUrl` 拼接审核结果参数，调用第三方地址并更新消息结果 |
| `orderStatusChange.service.queue` | `MQProducer.sendOrderStatusMessage` | `ActiveMqConsumer.orderStatusChange` | 订单状态变化后回调第三方机构，并写 `zhyf_mq_message` |
