# 旧系统业务全景

本文档用于梳理旧系统从机构接入、处方推送、订单履约、PDA 作业、煎煮、物流到回调补偿的完整业务链路。

## 1. 总体业务定位

旧系统围绕“医疗机构处方履约”展开，核心不是单纯订单 CRUD，而是一条从机构推方到药房履约、PDA/设备作业、物流出库、外部状态回调的业务链路。

第一轮代码阅读确认的主链路：

```text
机构/社康/医院
 -> 推送处方或由调度器拉取处方
 -> 创建订单、处方、处方明细
 -> 后台审核/处理
 -> PDA 扫码绑定煎煮设备
 -> PDA 上报煎煮状态
 -> 标签打印/包装
 -> 创建物流单或 PDA 出库
 -> 顺丰/EMS 物流轨迹回调
 -> 更新订单状态、写物流轨迹
 -> MQ 推送状态给外部系统
 -> 医院/机构查询订单状态
```

## 2. 核心参与系统

| 系统 | 在业务链路中的角色 |
| --- | --- |
| `zhyf-api` / `sc-zhyf-api` | 对外 API 入口，承接机构推单、PDA、物流、设备、智能云/医院定制接口 |
| `zhyfsupport` | 药房/煎煮中心后台，处理订单、处方、设备、物流、配置、权限、日志 |
| `sc-zhyf-communityhospital` | 社康调度器，从各区社康系统拉消息并上传订单 |
| `hospital-query` | 医院侧查单后台，提供订单、处方、物流查询 |
| `sc-address-input` | 患者地址补录与医院地址推送 |

## 3. 机构推单与订单创建

主要入口：

| 应用 | Controller | 接口 |
| --- | --- | --- |
| `zhyf-api` | `OrderApiController` | `/createOrder` |
| `zhyf-api` | `OrderApiController` | `/cancelOrder`、`/queryOrder`、`/updateOrderAddress`、`/getOrderId` |

`/createOrder` 已确认流程：

1. 解析机构请求 JSON 为 `CreateOrderParam`。
2. 从请求中记录创建 IP。
3. 校验 header 基础参数。
4. 取 `orderTime`、`tel`、处方号列表参与签名校验。
5. 校验订单和处方数据。
6. 按 `appKey + createIp` 做 IP 白名单校验。
7. 用 Redis `setNX` 对 `createOrder-{appKey}-{处方号列表}` 做 5 秒防重复提交。
8. 根据机构和医院处方号查询旧处方，防止有效处方重复提交。
9. 开启数据库事务。
10. 生成订单号。
11. 写订单主表。
12. 按处方列表写处方主表。
13. 按处方明细写处方明细表。
14. 写“煎药中心接单”物流/轨迹记录。
15. 提交事务。
16. 返回订单号和新系统内部处方号。
17. 发送创建订单成功 MQ 消息。

涉及旧表初稿：

| 表 | 作用 |
| --- | --- |
| `zhyf_order` | 订单主表 |
| `zhyf_prescription` | 处方主表 |
| `zhyf_prescription_details` | 处方明细 |
| `zhyf_institutions` | 机构配置、回调地址、存储类型等 |
| `zhyf_logis_info` | 订单轨迹/物流轨迹/履约日志 |
| `zhyf_log_exception` | 接口异常记录 |

重构要点：

- 旧系统把订单、处方、轨迹和消息发送揉在一个服务里；新系统应拆为订单域、处方域、履约轨迹、Outbox 事件。
- 旧系统事务提交后直接发 MQ；新系统应使用 `event_outbox` 保证事务和消息一致。
- 旧系统使用机构 `appKey` 作为公司编号；新系统应映射为 `tenant_app.app_key`，业务表保存 `tenant_id` 和 `institution_id`。

## 4. PDA 作业流程

主要入口：

| 接口 | 业务含义 |
| --- | --- |
| `/pdaUserLogin` | PDA 用户登录 |
| `/pdaGetDecoctEquip` | 查询煎煮设备 |
| `/pdaBindPrescription` | 处方绑定煎煮设备 |
| `/pdaOpDecoctStatus` | 开始、结束、取消、终止煎煮 |
| `/pdaGetCanOpRecipeList` | 查询可操作处方 |
| `/pdaLabelPrintInit` | 标签打印初始化 |
| `/pdaRecipeQuery` | 查询处方信息 |
| `/pdaLabelPrint` | 写标签打印记录 |
| `/pdaLogisticsOutbound` | PDA 物流出库 |

### 4.1 PDA 登录

已确认逻辑：

1. 校验 header。
2. 校验账号、密码参数。
3. 对 `account`、`password` 做签名校验。
4. 查询后台用户表。
5. 校验用户状态和密码。
6. 返回用户账号、姓名、模式、部门。

### 4.2 PDA 绑定处方与设备

`/pdaBindPrescription` 已确认流程：

1. 校验 header、必填参数和签名。
2. 根据 `recipeId` 查询处方。
3. 判断处方是否已绑定其他设备。
4. 查询设备是否存在。
5. 判断设备是否空闲。
6. 判断处方是否为代煎处方。
7. 开启事务。
8. 插入或更新处方绑定记录。
9. 更新设备状态为已使用。
10. 写 PDA 操作日志。
11. 提交事务。

涉及旧表初稿：

| 表 | 作用 |
| --- | --- |
| `zhyf_prescription` | 处方和煎煮状态 |
| `zhyf_prescription_bind_info` | 处方和设备绑定关系 |
| `zhyf_equipment_manage` | 煎煮设备 |
| `zhyf_log_info` | PDA 操作日志 |

### 4.3 PDA 煎煮状态上报

`/pdaOpDecoctStatus` 已确认要点：

- 校验处方、账号、操作状态、操作时间。
- `operStatus == 1` 时要求煎煮方案参数，如先煎、群煎、后下的温度和时长。
- 查询处方和绑定记录。
- 非代煎处方不允许煎煮操作。
- 写绑定状态、处方煎煮状态、履约轨迹和操作日志。

重构要点：

- 旧系统用数字 `operStatus` 表示煎煮状态，新系统应转换为明确枚举。
- 旧系统状态变更分散在 PDA Service、后台和物流逻辑中，新系统应收敛到煎煮状态机。
- 设备占用、处方绑定和煎煮开始必须放在事务边界内，并补并发控制。

## 5. 物流履约流程

主要入口：

| 接口 | 业务含义 |
| --- | --- |
| `/logistics/createLogisticsOrder` | 创建物流订单 |
| `/logistics/cancelLogisticsOrder` | 取消物流订单 |
| `/logistics/queryBillPrintInfo` | 查询面单 |
| `/logistics/printWaybills` | 打印面单预览 |
| `/logistics/queryLogisticsCost` | 查询物流费用 |
| `/logistics/queryEmsPdfFile` | 查询 EMS PDF 面单 |
| `/logistics/notify` | 顺丰路由回调 |

已确认物流下单要点：

1. 校验 header 和 body。
2. 校验订单号、寄件人、收件人、地址、支付方式、快递类型、包裹信息。
3. 做接口签名校验。
4. Redis 防重复。
5. 如果 `logisticsCompany == ems`，走 EMS 下单。
6. 否则走顺丰下单。
7. 本煎煮中心订单会根据订单信息、目的城市和配置选择月结卡号、代收款、快递产品。
8. 返回运单号。

已确认顺丰回调 `/logistics/notify` 要点：

1. 解析顺丰路由回调 JSON。
2. 用 `id + mailNo + orderId + opCode + secretKey` 做 SHA512 签名验证。
3. 如果是合并订单号，先查合并明细，再逐个原订单处理。
4. `opCode = 50 / 43 / 54` 时更新订单为已发货。
5. `opCode = 80 / 8000` 时更新订单为已签收。
6. 更新订单状态后发送订单状态 MQ。
7. 写物流轨迹。

涉及旧表初稿：

| 表 | 作用 |
| --- | --- |
| `zhyf_order` | 订单状态、物流状态、运单号等 |
| `zhyf_order_merge` | 合并订单与原订单关系 |
| `zhyf_logis_info` | 物流轨迹 |
| `zhyf_system_config` | 顺丰月结卡、物流配置 |
| `zhyf_logis_companycost_addr` | 物流费用地址规则 |
| `zhyf_logis_special` | 特殊物流规则 |

重构要点：

- 物流下单、轨迹接收、机构回调应拆开，避免一个 Controller 承担全部职责。
- 顺丰/EMS 应抽象为物流供应商 Adapter。
- 物流状态回调应先落库，再通过事件驱动更新订单和触发外部回调。

## 6. 社康调度流程

来源应用：`sc-zhyf-communityhospital`

已确认主任务：`AreaOrderTask`

流程：

```text
按区选择社康接口地址
 -> 拉取消息
 -> 识别上传订单消息
 -> 保存本地消息记录
 -> 根据机构、病人、门诊号获取订单详情
 -> 上传订单到智慧云煎药中心
 -> 失败写日志并短信通知
```

已确认消息字段：

| 字段 | 含义 |
| --- | --- |
| `XXLX` | 消息类型 |
| `JGID` | 机构 ID |
| `BRID` | 病人 ID |
| `MZXH` | 门诊序号 |
| `JZXH` | 就诊序号 |

重构要点：

- 社康调度是“拉模式”接入，不同于机构直接调用 `/createOrder`。
- 新系统应支持外部接入适配器，将拉取、转换、上传和补偿统一建模。
- 本地消息防重复和失败通知应升级为可观测的重试/死信/人工补偿。

## 10. 第二轮确认补充

### 10.1 订单状态链路

旧系统订单状态来自 `OrderStatusEnum`，是数字编码。订单表使用 `order_status` 保存当前状态，并用 `order_status_follow` 字符串追加状态流转历史。

已确认重要状态：

```text
5  审核中/已接单
15 调剂中
20 复核完成
25 煎煮中
27 煎煮完成
30 打包完成
36 已发货
37 运输中
40 签收/派送完毕
98 审核失败
99 用户取消
```

新系统不应继续用裸数字散落在业务代码里，应统一为枚举和状态机。

### 10.2 PDA 煎煮状态链路

PDA 绑定处方与设备后，`zhyf_prescription_bind_info.oper_status` 初始化为 `0`。

PDA 操作状态：

```text
0 已绑定，待煎煮
1 开始煎煮
2 煎煮完成
8 取消煎煮
9 终止煎煮
```

开始煎煮时：

- 更新绑定表 `oper_status=1` 和 `plan_value`。
- 更新处方 `boil_time_start`、`boil_status=1`。
- 更新订单状态为 `25`。
- 写履约轨迹“开始进行煎煮”。
- 写 PDA 操作日志。
- 写永安鑫煎煮机工作参数。

煎煮完成时：

- 更新绑定表 `oper_status=2`。
- 更新处方 `boil_time_end`、`boil_user_name`、`boil_status=2`。
- 更新订单状态为 `27`。
- 释放设备 `use_status=0`。
- 发送订单状态变化 MQ。
- 写履约轨迹“煎煮完成”。
- 写 PDA 操作日志。

### 10.3 物流回调状态链路

顺丰回调：

```text
opCode 50 / 43 / 54 -> 订单状态 36 已发货
opCode 80 / 8000    -> 订单状态 40 签收
其他且已发货       -> 订单状态 37 运输中
```

EMS 回调：

```text
opCode 203      -> 订单状态 36 已发货
opCode 748/704  -> 订单状态 40 签收
其他且已发货    -> 订单状态 37 运输中
```

顺丰按签名校验后处理；EMS 先对 `logitcsInterface` 做 SM4 解密，再按运单号找订单。

物流状态更新使用 `updateOrderLogisticsStatus`，SQL 限制为只允许 `order_status < 新状态` 时更新，避免状态倒退。

### 10.4 异步消息链路

创建订单成功后发送 ActiveMQ 队列：

```text
createOrderSuccess.service.queue
```

消费者 `ActiveMqConsumer.createOrderSuccessNotify` 做四步：

1. 药品目录匹配：`prescriptionHerbsService.matchHerbIndex(orderId)`。
2. 处方校验、楼层建议、煎煮机建议、位置校验：`orderValidateService`。
3. 调用审方平台并写审方信息：`reviewRecipeService.callRemoteUpdateReviewInfo`。
4. Redis 写 `push_order_flag`，有效期 30 分钟。

订单状态变化发送 ActiveMQ 队列：

```text
orderStatusChange.service.queue
```

消费者 `ActiveMqConsumer.orderStatusChange` 异步调用第三方机构回调地址，并写 `zhyf_mq_message` 记录结果。

### 10.5 旧系统消息可靠性问题

已确认旧系统存在以下问题：

- 业务事务提交后直接发 ActiveMQ，数据库事务和消息发送不是同一可靠边界。
- 第三方回调结果写 `zhyf_mq_message`，但没有形成统一 Outbox。
- 消费端没有明确的幂等表设计，部分依赖状态大小判断、消息记录和重试字段。
- 新系统需要用 `event_outbox`、`message_consume_log`、`callback_record` 替代旧的分散式消息记录。

## 11. 第三轮确认补充：后台履约链路

第三轮只读分析重点为 `zhyfsupport` 后台页面和 `sc-zhyf-support` 变体能力。

### 11.1 后台审核

入口：`OrderController.saveExamineOrderInfo`

已确认流程：

1. 查询订单，若 `order_status > 5` 则拒绝重复审核。
2. 按机构物流特殊规则和地址费用规则计算物流费。
3. 更新订单状态为 `15`，并追加 `order_status_follow ->15`。
4. 更新处方审核人、审核时间、审核结果。
5. 更新处方明细。
6. 根据机构配置发送审核通过短信。
7. 发送订单状态变化 MQ，状态为 `15`。
8. 写后台操作日志和履约轨迹。
9. 审核成功后释放订单锁。

业务含义：旧系统把审核通过后的订单视为进入调剂阶段，即状态 `15`。

### 11.2 调剂打单

入口：`OrderController.adjustPrescription`

已确认流程：

1. 根据处方号查询处方。
2. 若处方 `print_status=0`，校验操作员工号存在且 `type=1`。
3. 写处方 `dispense_user_id`、`dispense_time`、`print_status=1`。
4. 写操作日志。
5. 写履约轨迹“打单成功”。
6. 返回处方和处方明细用于打印。

注意：调剂打单本身不直接推进订单状态到 `20`；订单状态 `20` 由复核完成触发。

### 11.3 复核

入口：

- `OrderController.getReviewPrescription`
- `OrderController.reviewAdjustPrescription`
- `OrderController.reviewPrescription`

已确认规则：

- 复核查询要求订单未取消、订单状态不低于 `15` 且低于 `20`。
- 复核操作要求处方有重量信息。
- 复核员工号要求 `type=2`。
- 配送到个人地址时，手机号和省市区详细地址不能为空。
- 代煎处方要求加水量和加水桶号，并校验水桶 30 分钟内未被占用。

复核保存流程：

1. 代煎处方写水桶绑定记录，`bind_type=1`、`oper_status=0`。
2. 写处方 `recheck_user_id`、`recheck_time`、复核流程图片地址。
3. 若订单下所有处方均有调剂人和复核人，则订单状态更新为 `20`，追加 `order_status_follow ->20`。
4. 发送订单状态变化 MQ，状态为 `20`。
5. 写后台操作日志和履约轨迹。

注意：旧代码中 `reviewAdjustPrescription` 在复核界面补调剂人后也会发送 `20` 状态 MQ，但真正的订单状态更新仍以所有处方完成复核为准。新系统需要避免重复回调。

### 11.4 打包、面单和自提

后台物流页面入口：

- `LogisticsController.logisPrint`
- `LogisticsController.saveLogisPrint`
- `LogisticsController.saveSelfEnd`

打包/面单流程：

1. 打包前要求订单状态至少为 `20`。
2. 若没有运单号，调用物流接口创建顺丰/EMS 物流单。
3. 更新订单状态为 `30`，追加 `order_status_follow ->30`。
4. 写物流公司、运单号、包裹重量、包裹数、打包时间、付款方式。
5. 发送发货短信。
6. 写打包操作日志和履约轨迹。
7. 发送订单状态变化 MQ，状态为 `30`。
8. 查询面单信息并返回打印数据。

自提完成流程：

1. 更新包裹重量、包裹数、物流金额为 0。
2. 设置物流公司为保留机构，运单号为订单号。
3. 写打包时间、出库时间、签收时间。
4. 订单状态直接更新为 `40`，追加 `order_status_follow ->40`。
5. 写签收操作日志。

### 11.5 合并发货

入口：`LogisticsController.addOrderMerge`

已确认规则：

- 多个订单可生成合并单号，格式为 `HB` + 当前时间戳。
- 合并前要求每个订单存在且没有已有合并记录。
- 合并前要求多个订单收货信息一致，否则拒绝合并。
- 打包合并单时会遍历原订单更新运单信息和状态，并将合并记录 `end_status` 改为 `1`。
- 顺丰/EMS 回调收到合并运单时，也会按合并关系更新原订单状态。

### 11.6 后台状态修正

入口：

- `OrderController.initializePrescriptionStatus`
- `OrderController.cancelPrescriptionStatus`
- `OrderController.processOrder`
- `OrderController.orderReceipt`
- `OrderController.batchOrderReceipt`

已确认含义：

- 初始化处方：订单回到 `5`，处方作业状态初始化；若原状态为 `30`，会调用取消物流；同时将处方水桶/煎药机绑定记录置为无效，并释放煎药机。
- 取消处方：单个处方写 `audit_result=1`；若订单下处方全部取消，则订单更新为 `99`，但发送状态 MQ 时旧代码传 `98`。
- 订单走流程：后台手工补齐审核、调剂、复核、煎煮、签收等时间，订单直接到 `40`，并发送状态 MQ。
- 手工签收和批量签收：直接更新订单状态为 `40`，写操作日志和履约轨迹。

新系统需要把这些定义为“人工干预/管理修正”流程，和正常履约状态机分开授权、审计。

### 11.7 设备与煎药机建议

主系统 `zhyfsupport` 已确认能力：

- 水桶档案维护。
- 设备档案维护：设备类型包括煎煮机、PDA、打码机。
- 设备启停和使用状态维护。
- PDA 与打码机关系维护。
- 标签打印记录查询。

`sc-zhyf-support` 变体系统额外能力：

- 查询煎药机温度曲线：`checkTemperatureCurve`、`getTemperatureCurve`，数据来自 `zhyf_jyj_state`。
- 查询煎煮设备作业记录：`listDecoctDeviceOperate`，数据来自 `zhyf_jyj_work`。

异步校验中的煎药机建议：

- `OrderValidateService.prescriValidate` 当前默认代煎处方建议 1 楼，非代煎建议 2 楼。
- 当前默认 `jyj_decoction_plan=1`。
- 旧代码保留了按先煎、后下、附子剂量推导煎煮方案 `1-12` 的逻辑，但主流程中被注释未启用。

新系统处理建议：

- 设备档案、设备占用、设备作业记录应独立建模。
- 煎药方案建议可以先按旧系统默认规则实现，复杂药品规则作为后续开关能力。
- `zhyf_jyj_state`、`zhyf_jyj_work` 属于变体系统能力，应作为可选设备监控模块纳入重构范围评估。

## 12. 第四轮确认补充：外围接入链路

第四轮只读分析重点为社康定时拉单、医院查单、地址补录，以及 `sc-zhyf-api` 与主 `zhyf-api` 的差异。

### 12.1 社康定时拉单

来源应用：`sc-zhyf-communityhospital`

核心任务：

- `MultiThreadScheduler`
- `AreaOrderTask`
- `LocalMessageTask`
- `OrderService4CommunityHospitalImpl`
- `OrderService4DrugstoreImpl`

定时任务：

```text
深圳各区拉单任务：每天 6:00-23:00，每 2 分钟执行一次
本地未处理消息任务：每天 6:00-23:00，每 3 分钟执行一次
社康状态失败重试任务：每天 6:00-23:00，每 20 秒执行一次
```

覆盖区域：

- 光明新区
- 福田区
- 龙华区
- 龙岗区
- 罗湖区
- 南山区
- 宝安区
- 盐田区
- 坪山区
- 大鹏新区

主流程：

1. 按区域调用社康接口拉取消息。
2. 解析消息中的机构、患者、门诊号、就诊号。
3. 先保存消息到本地 `ch_message`，防止调用社康后远端消息状态变化导致漏单。
4. 根据机构、患者、门诊号调用社康接口获取订单详情。
5. 将社康订单转换成智慧药房 `/createOrder.do` 请求。
6. 推送成功后，本地消息置为已处理。
7. 调用社康接口回写订单已上传状态。
8. 回写失败时写 `ch_update`，由重试任务继续处理。

关键异常处理：

- 获取社康订单失败，记录日志并发送异常通知。
- 保存本地消息失败，提示需要人工排查漏单。
- 上传智慧药房成功但回写社康失败，保存失败记录等待重试。
- 社康返回订单已取消但智慧药房还未收到订单时，重置本地消息状态，允许后续再次处理。

### 12.2 社康订单转换到智慧药房

- `callbackUrl` 会拼接 `areaCode`、`companyId`、`outpatientNum`，用于后续状态回调社康。

- 支持一个社康订单拆成多个智慧药房处方请求。

### 12.3 社康状态回写与退费

来源：`OrderService4CommunityHospitalImpl`

- `refund`：回写社康退费订单状态。

### 12.4 医院查单系统

来源应用：`hospital-query`

系统类型：基于 RuoYi / Shiro 的医院侧查询后台。

核心入口：`com.zhyf.hospital.controller.OrderController`

已确认功能：

- `/hospital/orderQuery`：订单查询页面。
- `/hospital/list`：订单列表数据。
- `/hospital/orderDetail`：订单详情页面。

- `/hospital/getLogisticsInfo`：物流轨迹页面。

页面入口：

| 入口 | 医院/场景 | 主要协议 |

### 12.6 `sc-zhyf-api` 与主 `zhyf-api` 差异

对比结果：

- Controller 文件名和核心接口基本一致。
- `SmartCloudOrderApiController`、`SzlgzxyyOrderApiController`、`MachineApiController` 在主版本与社康版本中均存在。
- 主要差异集中在少量实现细节。

已确认差异：

| 文件 | 差异 |
| --- | --- |
| `OrderServiceImpl` | 取消订单日志变量命名和日志输出略有差异，业务语义基本一致 |
| `LogisticsServiceImpl` | 物流下单默认 `expressType` 不同，主版本和社康版本分别使用不同默认值 |
| `MachineServiceImpl` | 代码格式差异为主，业务语义基本一致 |
| `ActiveMqConsumer` | 社康版本在解析消息前增加 JSON 格式校验，并在订单状态回调前输出调试信息 |

`sc-zhyf-api` 关键专属/定制能力：

- `SmartCloudOrderApiController`：智慧云格式订单同步、修改、取消、状态回调。
- `SzlgzxyyOrderApiController`：深圳龙岗中心医院 XML/RSA 接口。
- `MachineApiController`：煎药机/设备相关接口，包括水桶加水、加水完成、取方、煎煮回调、煎煮状态通知。

新系统建议：

- 不应把 `sc-zhyf-api` 当成完全独立业务线重写。
- 应先以主 `zhyf-api` 为标准接口模型，再把智慧云、龙岗中心医院、设备接口作为机构/设备适配器。
- 默认物流产品、签名兼容、JSON 校验等差异需要做配置化。
