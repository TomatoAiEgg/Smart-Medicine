# 旧系统问题清单

本文档记录旧系统在业务边界、状态管理、数据模型、消息可靠性、回调补偿、部署和可维护性方面的问题，并给出新系统重构处理原则。

## 1. 总体问题

| 问题 | 旧系统表现 | 重构处理原则 |
| --- | --- | --- |
| 多应用共享数据库 | `zhyfsupport`、`zhyf-api`、`zhyf-async`、社康、医院查单等围绕同一套核心库协作 | 新系统按领域建模，禁止跨服务直接改对方表 |
| 业务边界不清 | 订单、处方、PDA、物流、回调逻辑散落在 Controller、Service、定时任务、MQ 消费者中 | 建立订单、处方、煎煮、物流、回调、消息、机构等领域边界 |
| 状态推进分散 | 订单状态在后台、PDA、物流回调、人工修正中直接修改 | 使用状态机统一校验合法跳转，所有变更写状态日志 |
| 协议差异硬编码 | 不同医院、社康、物流、地址补录协议通过分支逻辑散落实现 | 使用机构适配器、地址适配器、物流适配器隔离差异 |
| 测试和观测不足 | 缺少稳定自动化测试、压测、链路追踪和消息堆积监控 | 第一版同步建设联调用例、压测脚本、指标和日志 |

## 2. 数据模型问题

| 问题 | 影响 | 新系统策略 |
| --- | --- | --- |
| 表命名和职责混杂 | 订单、处方、物流、消息、日志表存在历史字段和非规范字段 | 新表按领域拆分，旧字段通过映射文档保留兼容关系 |
| 缺少租户边界 | 旧系统更多依赖机构编号和用户可见范围控制 | 所有核心业务表必须带 `tenant_id`，机构编号作为业务标识 |
| 大表增长无明确治理 | 物流轨迹、日志、MQ 消息等表增长快 | 高增长表按月分区，建立时间、租户、状态组合索引 |
| 状态字段语义依赖代码 | 旧状态值为数字，含义分散在枚举、SQL 和页面中 | 新系统使用枚举编码和中文说明，保留旧状态映射 |
| 回调和消息记录混用 | `zhyf_mq_message` 同时承担异步结果、回调结果、重发记录 | 拆分为 `event_outbox`、`message_consume_log`、`callback_record` |

## 3. 消息可靠性问题

| 问题 | 影响 | 新系统策略 |
| --- | --- | --- |
| ActiveMQ 发送和业务事务不一致 | 业务成功但消息丢失、消息成功但业务回滚都可能发生 | 使用 Outbox，本地事务同时写业务表和事件表 |
| 消费幂等不统一 | 重复消费依赖局部判断 | 每个消费者记录 `message_consume_log`，按事件 ID 幂等 |
| 消息类型和业务动作耦合 | 队列名表达旧实现，不表达领域事件 | RocketMQ 使用 Topic + Tag 表达业务事件 |
| 死信和人工重放不足 | 回调失败、消费失败排查依赖日志和人工 SQL | 建立死信记录、重试策略、后台重放入口 |

## 4. 订单履约问题

| 问题 | 影响 | 新系统策略 |
| --- | --- | --- |
| 审核、调剂、复核、煎煮状态互相影响但缺少统一模型 | 容易出现订单状态和处方状态不一致 | 建立订单状态机、处方状态机、煎煮状态机，并定义聚合规则 |
| 人工修正能力缺少边界 | 后台可直接推进到完成或回退状态 | 保留必要人工修正，但必须写原因、操作人、前后状态和审计日志 |
| 重复推单处理依赖处方号等局部规则 | 不同机构的唯一性规则可能不同 | 按租户、机构、外部订单号、外部处方号建立幂等键 |
| 物流签收和机构回调耦合 | 物流状态变化直接触发旧回调消息 | 新系统通过物流事件驱动 callback-service，失败可补偿 |

## 5. PDA / MES / 设备问题

| 问题 | 影响 | 新系统策略 |
| --- | --- | --- |
| PDA、煎药机、水桶、打印机关系分散 | 设备占用释放、作业状态难追踪 | 统一设备、绑定、作业记录和设备日志模型 |
| 无真实设备时难联调 | 开发和测试依赖现场设备 | 第一版建设 PDA / MES 模拟器，覆盖重复扫码、乱序、超时、并发 |
| 设备差异侵入业务代码 | 不同设备接口细节可能影响订单主流程 | 使用 `MesAdapter`、`PdaAdapter` 隔离真实设备接入 |

## 6. 外围接入问题

| 问题 | 影响 | 新系统策略 |
| --- | --- | --- |
| 社康定时任务和主系统边界弱 | 拉单、上传、回写、失败重试分散 | 统一为集成任务、集成消息和回调记录 |
| 医院查单依赖旧权限字段 | 用户可见机构通过 `companyNums` 控制 | 新系统转为租户、机构、角色权限模型，兼容旧可见范围 |
| 地址补录多协议散落 | 每个医院页面、签名、加密、回推逻辑不同 | 地址补录进入独立适配层，核心只接收标准地址变更事件 |

## 7. 部署运维问题

| 问题 | 影响 | 新系统策略 |
| --- | --- | --- |
| 旧配置散落 | 数据库、Redis、MQ、密钥配置分散且可能明文出现 | 通过环境变量、配置中心或本地未提交文件注入敏感配置 |
| 单机部署特征明显 | Tomcat、Redis、ActiveMQ 等偏传统部署 | 新系统按 Docker、健康检查、监控指标和日志采集建设 |
| 本项目和其他项目资源容易混用 | 云服务器已有其他项目数据库、Redis 容器 | 本项目容器、网络、端口统一 `zhyf-` 命名，禁止误连 |

## 8. 第一版必须规避的风险

- 不要直接照搬旧 MySQL 表结构到 PostgreSQL。
- 不要把旧数字状态散落到各服务里直接赋值。
- 不要让设备模拟器成为临时脚本，必须纳入可重复联调能力。
- 不要把医院、社康、物流差异硬编码进订单主流程。
- 不要先做 AI 功能，必须先完整跑通旧功能替代闭环。
- 不要只做接口 Demo，必须从推单到回调、日志、消息、补偿全部闭环。
