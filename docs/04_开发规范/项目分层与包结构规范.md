# 项目分层与包结构规范

本文档定义本项目后端各模块的推荐分层方式。目标不是套老式三层，而是让代码按职责清晰分开，后续能稳步扩展。

具体服务边界、落地顺序和过渡规则以 `docs/03_架构设计/重构方案.md` 为准。

## 0. 参考结论

调研 Spring Boot / Java 生态中较流行的项目和架构样例后，当前结论如下：

- Spring 官方样例更偏“清晰、简单、少抽象”，不鼓励一开始就把项目拆得过度复杂。
- Spring Modulith 的核心思路是按业务模块组织代码，业务模块作为主包下的直接子包。
- DDD / Hexagonal / Clean Architecture 高星样例普遍强调领域规则和基础设施解耦，但实现方式差异很大，不能机械照搬。
- JHipster 这类生成器适合快速搭生产骨架，但生成代码偏重，不适合本项目当前手工重构阶段直接套用。

所以本项目采用：

```text
微服务边界按业务领域划分
微服务内部优先按业务能力组织
复杂能力内部再使用 application / domain / infrastructure
外部系统接入使用 adapter / port 思路
```

不采用：

```text
全项目 controller / service / repository 横向大分包
所有模块强行 Clean Architecture 六七层
一开始引入 Spring Modulith / JHipster 生成体系
```

## 1. 总原则

- 按业务能力分模块，不按技术名词堆类。
- 微服务内部优先按业务能力分包；业务能力复杂后，再在能力包内拆层。
- 控制器只负责协议转换，不写业务判断。
- 领域规则尽量放在领域对象或领域服务中。
- 数据库、RocketMQ、HTTP 客户端、第三方 SDK 都放在基础设施层。
- 小服务可以先简化，但包边界要保留。

## 2. 推荐组织方式

### 2.1 优先：按业务能力组织

适合大多数业务服务。

示例：

```text
com.zhyf.order
├── create
├── query
├── cancel
├── address
├── status
└── infrastructure
```

每个业务能力内部可以继续拆：

```text
create
├── api
├── application
├── domain
└── infrastructure
```

### 2.2 允许：按技术入口组织

适合 `message-service` 这种技术职责明显的服务。

示例：

```text
com.zhyf.message
├── consumer
├── publisher
├── application
├── domain
└── infrastructure
```

### 2.3 避免：全局横向三层

不建议：

```text
controller
service
repository
entity
dto
```

原因是业务复杂后，所有用例都会堆到同一批包里，旧系统的问题会重现。

## 3. 推荐层次

### 3.1 接口层

职责：

- 接收 HTTP / MQ / 定时任务 / CLI 入口
- 参数校验
- DTO 转换
- 返回统一响应

建议包名：

```text
api
consumer
job
config
```

### 3.2 应用层

职责：

- 编排用例
- 事务边界
- 协调领域对象和仓储
- 处理幂等、状态流转、流程推进

建议包名：

```text
application
```

### 3.3 领域层

职责：

- 核心业务对象
- 值对象
- 业务规则
- 状态判断

建议包名：

```text
domain
```

### 3.4 基础设施层

职责：

- JDBC / JPA / MyBatis 持久化
- RocketMQ Producer / Consumer
- HTTP Client
- 外部系统适配
- 运行时配置

建议包名：

```text
infrastructure
```

## 4. 微服务包结构模板

### 4.1 `order-service`

```text
com.zhyf.order
├── create
├── query
├── cancel
├── address
├── status
├── infrastructure
└── OrderServiceApplication
```

当前第一阶段代码量小，可以暂时保持：

- `api`
  - `InstitutionOrderController`
  - `GlobalExceptionHandler`
- `application`
  - `OrderService`
  - `OrderCreateCommand`
  - `OrderCreateResult`
- `domain`
  - `InstitutionApp`
  - `OrderSnapshot`
  - 后续可放状态机、领域事件
- `infrastructure`
  - `OrderRepository`
  - 后续可放消息/外部适配

当 `order-service` 继续增加订单取消、地址修改、状态流转后，再迁移成按能力拆包：

```text
create
├── api
├── application
└── domain

status
├── application
└── domain

infrastructure
└── persistence
```

### 4.2 `message-service`

```text
com.zhyf.message
├── consumer
├── publisher
├── application
├── domain
├── infrastructure
└── MessageServiceApplication
```

建议拆分：

- `consumer`
  - RocketMQ 消费入口
  - 事件路由
  - 幂等协调
- `publisher`
  - Outbox 发布器
  - RocketMQ 发送器
- `application`
  - `OrderCreatedHandler`
  - 后续订单/处方/回调处理用例
- `domain`
  - `MessageEvent`
  - `OutboxEvent`
- `infrastructure`
  - `OutboxRepository`
  - `MessageConsumeRepository`
  - `OrderValidationRecordRepository`
  - `RocketMqProperties`

## 5. 当前项目的现实做法

当前项目规模还不大，不必一次性把包切得特别碎。

建议按下面的节奏演进：

1. 先保证每个微服务有清晰边界。
2. 业务服务优先按业务能力拆包。
3. 技术服务可以按技术入口拆包。
4. 当某个业务能力里代码超过一屏、规则变复杂，再拆 `application / domain / infrastructure`。
5. 当领域规则复杂起来，再在 `domain` 下加 `service`、`policy`、`event`、`valueobject`。

## 6. 不建议的写法

- 所有类平铺在同一个包里。
- 控制器直接写 SQL。
- 消费者直接改业务表，没有应用层编排。
- 领域对象只放字段，不放规则。
- 工具类替代应用层。
- 把 MQ 发布和业务更新揉在一个大方法里。

## 7. 对 `message-service` 的具体建议

`message-service` 现在最适合的定位不是“业务微服务”，而是“事件编排与消息基础设施服务”。

建议职责分成三块：

- Outbox 发布
- RocketMQ 消费
- 消费结果落库与幂等

后续再逐步增加：

- 订单创建后校验
- 审方请求
- 回调请求
- 通知请求

## 8. 对 `order-service` 的具体建议

`order-service` 现在已经具备应用层雏形。

建议保持：

- `api` 只管协议
- `application` 只管推单和查询用例
- `domain` 只放机构应用、订单快照、状态判断
- `infrastructure` 只管 JDBC 和数据映射

这样后面补订单取消、地址修改、状态流转时不会把代码继续堆平。

## 9. 结论

本项目不走传统“控制器-服务-DAO”平铺结构，而走“业务能力优先 + 必要处四层结构 + 外部适配隔离”的方式。

这样做的目的只有一个：

- 既能让现在这版快跑
- 又不会把后面审方、物流、回调、设备这些复杂流程写散
