# 开发变更与编码规范总则

## 1. 目的

本文件是项目后续所有开发变更的统一需求入口。

任何新的编码、重构、字段调整、接口调整、部署调整、脚本行为调整、页面调整，都必须先写方案，再等确认，再落地。

## 2. 生效范围

适用于以下内容：

1. 后端接口。
2. 数据库表结构。
3. 手机自动化脚本。
4. 后台管理页面。
5. 部署脚本与环境变量。
6. 文档、规范、项目记录。

## 3. 变更流程

所有变更必须按以下顺序执行：

1. 先写方案。
2. 方案里必须说明现状、目标、变更范围、不变范围、风险、落地顺序。
3. 把方案发给我确认。
4. 只有在我明确同意后，才能开始编码或改文件。
5. 落地后要补充验证结果。
6. 如果是长期规则，要同步更新到本文件。

## 4. 编码要求

1. 先读现有代码和现有文档，再写代码。
2. 优先复用现有模型、表、接口和页面。
3. 不得为了局部需求随意新增系统级抽象。
4. 数据结构变更必须同时给出 migration。
5. 接口变更必须同步更新类型定义。
6. 手机脚本改动必须保留可审计日志。
7. 真正会影响设备行为的改动，必须有明确的安全边界。

## 5. 文档要求

1. 需求、规范、设计方案放 `docs/当前需求`。
2. 会议记录、联调记录、阶段总结、临时分析放 `docs/项目记录`。
3. 新需求如果会影响实现，必须先形成方案文档。
4. 历史方案如果不再作为当前主线，要标明废弃或暂停，不得直接删除上下文。

## 6. 当前统一命名要求

1. 设备对外统一使用 `deviceCode`。
2. 手机本地 token 统一使用 `deviceToken`。
3. 设备注册后的唯一绑定关系必须可追踪。
4. 后台查询设备优先按 `deviceCode`。
5. 接口与类型中不要同时制造多个含义相近但用途不同的主标识，除非文档中明确区分。

## 7. 数据库要求

1. 所有字段新增必须写 migration。
2. 默认值必须写清楚。
3. 关键字段要考虑唯一约束或索引。
4. JSON 字段只用于复杂扩展，不用于替代明显的一层字段。
5. 设备画像、任务配置、审计记录要分层存放。

## 8. 手机脚本要求

1. 本地配置优先，后台配置次之，默认值兜底。
2. 任务执行必须可暂停、可停止、可审计。
3. 真正会发送或发布的动作必须保留本地确认开关。
4. 任何新增自动行为都必须先有模式开关，不能默认无条件启用。

## 9. 后台页面要求

1. 后台页面优先服务当前要用的工作流，不做空壳总览页。
2. 表单和表格字段要和实际数据库字段一致。
3. 保存后如果需要手机立即生效，要明确触发刷新命令。

## 10. 部署要求

1. 数据库结构变更和应用部署分开看待。
2. 涉及 schema 的变更，部署前必须确认 migration 执行方式。
3. 生产部署脚本和本地脚本都要保留。
4. 影响真机运行的变更，必须先确认测试设备和回滚方式。

## 11. 需要我确认的边界

以下情况必须先出方案，不得直接改：

1. 新增大模块。
2. 改变设备注册流程。
3. 改变默认执行模式。
4. 改动数据库主键、唯一键、绑定关系。
5. 改动手机端真实发送、自动互动、风控开关。
6. 改动部署方式或环境变量。

## 12. 维护规则

1. 以后新的编码规范要求，优先补到本文件。
2. 新的流程约束，先补总则，再补专项方案。
3. 如果专项方案和总则冲突，以总则为准，除非我明确要求覆盖。

## 13. 协作与代码提交规则

1. 未收到明确指令，不得主动 `git push`。
2. 用户只要求“继续”“检查”“验证”时，默认只做本地检查、修复和记录，不自动推送远程仓库。
3. 推送远程仓库必须明确目标仓库、分支和提交内容；如果涉及多个仓库，必须逐个确认推送结果。
4. 工作区存在其他人或其他窗口产生的改动时，只能修改本次任务相关文件，不得回滚、覆盖或格式化无关文件。
5. 发现无关文件被删除、移动或修改时，只记录状态，不主动恢复，除非用户明确要求。
6. 操作服务器前要考虑其他窗口可能正在操作同一服务，避免同时重启、清库、覆盖部署造成冲突。

## 14. 变更前检查规则

1. 修复问题前必须先检查脚本结构、历史遗留、版本兼容和运行环境，不得只看表面报错直接改。
2. APK 问题要同时检查四类信息：
   - 当前安装包包名、版本号、versionCode。
   - 手机上是否存在旧 AutoX、旧脚本、七彩虹或历史公司脚本。
   - 无障碍、悬浮窗、USB 调试、MIUI 权限是否实际生效。
   - 后台是否收到注册、心跳、日志和命令轮询。
3. 后台出现多设备、重复设备或陌生设备时，优先排查设备注册逻辑、设备唯一标识、token 绑定和历史脏数据。
4. 手机端显示“运行中”但后台没有设备时，要先看手机本地日志和 API 访问日志，再判断是手机问题、网络问题还是后台注册问题。

## 15. 手机 APK 与脚本运行规范

1. APK 必须有简单可操作页面，至少提供：
   - 检查权限。
   - 打开无障碍设置。
   - 打开悬浮窗设置。
   - 启动采集脚本。
   - 停止采集脚本。
   - 启动守护脚本。
   - 关闭全部脚本。
   - 刷新状态。
2. 页面必须显示当前版本、无障碍状态、悬浮窗状态、采集脚本状态和守护脚本状态，不能只提示“已启动”。
3. 当前项目 APK 不应依赖旧 AutoX.js v6、七彩虹或其他历史脚本包。
4. 打包后必须记录 APK 路径、包名、versionName、versionCode、大小和 hash。
5. 真机验证 App 启动时，不要使用 `adb am start -S` 或 force-stop。该操作会强制停止 App，容易导致 AutoJs 无障碍服务显示已开启但实际未运行。
6. 如果出现“无障碍服务已启用但未运行”，应到系统无障碍设置里关闭再开启 AgriVideoCollector，必要时重启手机。
7. 覆盖安装 APK 后必须重新检查 MIUI 权限，特别是无障碍、悬浮窗、USB 安装、后台运行权限。

## 16. 设备注册与鉴权规范

1. 设备必须唯一注册，不允许同一台设备因脚本重启或重复安装生成多条有效设备。
2. 设备对外主标识使用 `deviceCode`，设备本地密钥使用 `deviceToken`。
3. 设备名称不能在脚本中写死为 `Android001` 这类固定值，应由后台根据 `deviceCode` 生成默认名称，并支持后台修改显示名称。
4. 首次注册时允许手机提交 `deviceToken` 进行绑定；绑定后普通业务请求不应继续明文发送 token。
5. 普通移动端请求应使用轻量签名，至少包含：
   - `X-Device-Id`
   - `X-Timestamp`
   - `X-Body-SHA256`
   - `X-Signature`
6. 后台根据 `deviceCode` 查询绑定 token 后验签；token 不匹配、签名错误、时间戳异常时必须拒绝请求。
7. 命令拉取和命令回执必须绑定当前设备，不能让一台设备领取或确认另一台设备的命令。
8. 后台设备列表不返回 token 明文，只能展示是否已绑定 token。
9. 后续安全升级方向包括 nonce 防重放、HTTPS、token hash 存储、首次注册审核和请求限流。

## 17. 数据清理与生产数据规范

1. 清库、清日志、清采集记录、清心跳、清命令都属于高风险操作，必须有用户明确指令。
2. 清理运行期脏数据时，默认只清理：
   - 采集记录。
   - 运行日志。
   - 心跳流水。
   - 待执行和历史命令。
3. 设备表、设备 token、设备名称、任务配置默认保留，除非用户明确要求删除。
4. 清理后必须记录清理范围和保留范围，避免后续误判设备丢失。
5. 线上数据库字段缺失时，可以做热修，但必须同步记录缺失字段、执行 SQL 和后续 migration 补齐要求。

## 18. 日志与联调记录规范

1. 手机端必须保留本地运行日志，方便没有后台或后台异常时排查。
2. 当前手机本地日志推荐路径：

```text
/sdcard/AgriVideoCollector/datasource/运行日志/YYYY-MM-DD.log
```

3. 运行日志上报字段必须结构化，`level`、`message`、`context`、`deviceId` 类型要稳定。
4. 后台接口验证要同时看 HTTP 状态码、业务错误码、服务端日志和手机本地日志。
5. 每次 APK 真机验证后，要把版本、设备、后台状态、已验证接口和遗留风险写入 `docs/项目记录`。
6. 完成的重要功能必须维护到 `docs/项目记录/项目记录.md` 或对应项目记录文件。

## 19. 自动行为与安全边界

1. 默认不打开自动评论、点赞、关注、私信、加群、发布等真实互动行为。
2. 任何真实互动能力必须有独立开关、测试设备范围、审计日志、失败原因和人工确认路径。
3. 只读采集、评论读取、计划生成和真实执行必须分层，不允许把识别逻辑和真实发送混在一个不可审计流程里。
4. P0 阶段优先保证可观察、可停止、可回滚，不追求大规模并发。
5. 不绕过验证码、登录校验或平台安全校验；遇到风险页面应停止或等待人工处理。
