# 架构总蓝图

本文档是本项目后续所有架构、规格、模块、数据库、流程、包结构设计的总前提。先定边界，再写功能。

## 1. 设计顺序

本项目采用固定顺序：

```text
业务域
 -> 服务边界
 -> 数据边界
 -> 事件边界
 -> 包结构
 -> 接口与流程
 -> 数据表
 -> 编码
```

后续任何功能文档都必须先对齐这里的边界，再展开细节。

## 2. 业务域划分

第一批核心业务域：

- 机构接入
- 订单
- 处方
- 消息与事件
- 运维审计

第二批扩展业务域：

- 审核调剂复核
- 煎煮与设备
- 物流与签收
- 回调
- 社康集成
- 医院查单
- 地址补录

第三批增强业务域：

- 报表
- 监控
- AI 提效

## 3. 服务边界

### 3.1 核心服务

```text
gateway
auth-institution
order-service
message-service
workflow-service
ops-service
admin-web
```

说明：

- `gateway` 负责统一入口、签名、白名单、限流、访问日志。
- `auth-institution` 负责租户、机构、应用密钥、白名单、回调配置。
- `order-service` 负责订单主链路、订单状态机、状态日志。
- `message-service` 负责 Outbox 发布、RocketMQ 消费、消费幂等、基础校验结果处理。
- `workflow-service` 负责审核、调剂、复核、工作任务和人工流程控制。
- `ops-service` 负责日志、审计、异常、联调记录。
- `admin-web` 负责后台管理入口。

### 3.2 扩展服务

```text
prescription-service
decoction-service
logistics-service
callback-service
integration-service
device-service
portal-service
report-service
```

说明：

- 这些服务按总体架构预留，随履约、外部系统和统计能力推进逐步落地。
- 不用为追求模块数量提前把未稳定的业务拆空。
- 目标是把职责拆清楚，不是把模块数量堆满。

## 4. 分阶段实施与目标架构的关系

项目按阶段交付，但架构不按阶段分叉。阶段一、阶段二、阶段三只是实施顺序，所有过渡实现都必须服务于同一套目标架构。

允许的过渡：

- `order-service` 先承接订单创建和基础处方落库，后续再拆出 `prescription-service`。
- `message-service` 先处理基础校验和事件发布，后续再按事件用途拆分更多处理器。
- 基础设施表可以先由现有承载模块的 migration 创建，后续再集中迁移。

不允许的过渡：

- 以“先快点做出来”为理由长期混用服务职责。
- 把外部适配、业务规则、DB 访问、消息处理全部堆在一个包里。
- 没有目标边界，只做临时拼接。

## 5. 分层规则

每个服务内部遵循：

- `api`：HTTP / MQ / 定时任务入口
- `application`：用例编排、事务边界、流程推进
- `domain`：业务规则、状态、值对象
- `infrastructure`：JDBC、RocketMQ、外部接口、配置

技术型服务可进一步使用：

- `consumer`
- `publisher`
- `job`
- `adapter`

## 6. 数据边界

原则：

- 一个核心业务域对应一组主表、明细表、状态表、日志表。
- 事件和幂等表属于平台级公共基础设施。
- 外部协议数据和内部领域数据必须分开保存。

当前阶段优先级：

1. 机构接入数据
2. 订单与处方数据
3. Outbox 与消费日志
4. 审计和访问日志

后续再扩展：

- 设备与煎煮
- 物流
- 回调
- 社康与地址

## 7. 事件边界

事件只负责传递业务事实，不承担业务判断。

核心事件原则：

- `ORDER_CREATED` 是订单创建事实。
- 发布必须经 Outbox。
- 消费必须有幂等日志。
- 消费结果必须可追踪。

事件设计遵循：

- 事件名稳定
- payload 可扩展
- 发布和落库同事务
- 消费失败可重试

## 8. 包结构边界

推荐做法：

- 业务服务按能力拆包。
- `message-service` 按技术职责拆包。
- `gateway` 按过滤器、签名、路由、日志拆包。

不推荐：

- 单包大杂烩。
- 过早搞过多通用框架层。
- 先造抽象，再找业务填。
