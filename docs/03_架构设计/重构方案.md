# 智能药房 SaaS 平台重构方案

本文档是 `docs/03_架构设计` 的唯一总方案入口，合并原 `架构总蓝图.md`、`总体架构落地设计.md` 和旧 `重构方案.md` 中重复的总体架构内容。

专项细节仍分别查看：

- `业务流程.md`：旧系统业务流程和状态机总结，本次只作为参考，不改动。
- `数据库设计.md`：表结构、字段和迁移依据。
- `RocketMQ事件设计.md`：Topic、事件、消费和补偿设计。
- `PDA与MES模拟方案.md`：PDA/MES 模拟、旧协议兼容和设备联调设计。

## 1. 项目定位

新系统目标是用一套多租户 SaaS 平台替代旧智慧药房后台、机构接口、医院查单、地址补录、PDA/MES 作业、物流回调、社康外围集成和运维补偿能力。

第一版不是做一个缩小 Demo，而是先替代旧系统的核心生产闭环：

```text
机构推单
 -> 订单/处方落库
 -> 审核
 -> 调剂
 -> 复核
 -> 煎煮
 -> 打包/物流/签收
 -> 回调补偿
 -> 查单/报表/运维排错
```

社康配置化拉单和更完整的社康区域适配暂缓，不进入当前核心闭环；`integration-service` 保留最小消息归档、回推、查单和重试基础。

## 2. 重构原则

- 先替代旧系统核心业务，再做增强能力。
- 阶段只表示实施顺序，不拆出另一套阶段架构。
- 外部协议优先兼容旧系统，内部实现按领域服务、状态机、Outbox 和事件驱动重构。
- 订单、处方、审核、调剂、复核、煎煮、物流、回调必须可审计、可追踪、可补偿。
- 外部系统差异通过 adapter 或独立服务隔离，不写死在订单主链路里。
- 管理端、运维端、模拟器都基于同一套后端能力，不重复造业务逻辑。
- AI、复杂 BI、大屏和自动化调度不进入第一版主链路。

## 3. 总体架构

```text
外部机构 / 医院门户 / PDA / MES / 物流 / 社康外围 / admin-web
        |
      gateway
        |
        +-- auth-institution
        +-- order-service
        +-- prescription-service
        +-- workflow-service
        +-- message-service
        +-- decoction-service
        +-- device-service
        +-- logistics-service
        +-- callback-service
        +-- integration-service
        +-- portal-service
        +-- ops-service
        +-- report-service
        |
 PostgreSQL + Redis + RocketMQ + 观测组件
```

设计顺序固定为：

```text
业务域
 -> 服务边界
 -> 数据边界
 -> 事件边界
 -> 包结构
 -> 接口与流程
 -> 数据表
 -> 编码
```

## 4. 服务边界

| 服务 | 目标职责 | 当前落地状态 |
| --- | --- | --- |
| `gateway` | 统一入口、路由、机构签名校验、IP 白名单、限流、访问日志 | 已承接机构推单入口转发和访问日志 |
| `auth-institution` | 租户、机构、应用密钥、白名单、回调配置、后台用户权限基础 | 已承接机构应用内部查询 |
| `order-service` | 机构推单、订单/处方基础落库、订单状态机、订单状态日志、订单查询、订单履约进度、内部状态推进 | 已落地核心订单链路、履约进度和状态日志 |
| `prescription-service` | 处方、处方明细、处方修改、审方摘要、处方状态机 | 当前为模块骨架，处方表暂由 `order-service` 承载 |
| `workflow-service` | 后台审核、调剂、复核、订单锁、人工流程控制、工作任务和状态推进 | 已承接 `ORDER_REVIEW`、`PRESCRIPTION_DISPENSE`、`PRESCRIPTION_RECHECK`，调剂写 `dispense_record` |
| `message-service` | Outbox 扫描发布、RocketMQ 生产、消费入口、消费幂等、消息补偿 | 已承接 `ORDER_CREATED` 发布和基础校验记录 |
| `decoction-service` | PDA 作业、扫码绑定、水桶/设备、加水、煎煮、MES 适配 | 已落地 PDA/MES 模拟器、旧协议兼容入口、作业明细和 adapter 抽象 |
| `device-service` | 设备档案、设备占用、设备关系、设备状态 | 目标服务，当前可随煎煮域分阶段拆分 |
| `logistics-service` | 打包出库、面单、物流轨迹、签收、自提、顺丰/EMS 适配 | 已落地待打包、打包、发货、签收、轨迹和旧回调兼容入口 |
| `callback-service` | 机构状态回调、物流回调补偿、失败重试、死信、人工重放 | 已落地回调记录、真实 HTTP 发送、手动派发、失败重试和 `DEAD` 终态 |
| `integration-service` | 社康、医院查单、地址补录等外围协议适配 | 已落地消息归档、社康状态回写、地址回推、医院处方查单和重试任务；社康完整拉单暂缓 |
| `portal-service` | 医院查单、地址补录等外部轻量门户 | 已落地订单查单和地址补录申请最小入口 |
| `ops-service` | 操作日志、异常日志、审计、链路追踪、监控聚合、联调排错 | 已落地日志查询、回调/集成失败聚合和核心健康概览 |
| `report-service` | 报表、统计、导出、经营分析 | 已落地核心只读统计和 CSV 导出 |
| `admin-web` | 管理端订单、处方、审核、调剂、复核、煎煮、物流、报表、运维页面 | 已有第一版入口，但还未完全还原旧平台后台核心操作体验 |

服务拆分规则：

- 服务边界按业务归属决定，不按当前代码方便程度决定。
- 过渡期可以由已落地服务承载表结构或接口，但逻辑归属必须以本表为准。
- 跨服务状态推进优先通过应用服务接口或事件，不允许直接跨服务改库。
- 不为追求模块数量提前把未稳定的业务拆空。

## 5. 数据边界

| 数据 | 逻辑归属 | 落地策略 |
| --- | --- | --- |
| `tenant`、`institution`、`institution_app`、`institution_ip_whitelist` | `auth-institution` | 机构、应用密钥和白名单基础数据 |
| `order_main`、`order_status_log` | `order-service` | 订单主表和状态日志，所有状态推进必须写日志 |
| `prescription`、`prescription_detail` | `prescription-service` | 当前由 `order-service` migration 创建，后续迁移到处方域维护 |
| `workflow_task`、`dispense_record` | `workflow-service` | 审核、调剂、复核任务和调剂记录 |
| `order_validation_record` | `message-service` / `workflow-service` 读取 | 订单事件基础校验结果 |
| `event_outbox` | 平台事件基础设施 | 业务服务写入，`message-service` 发布 |
| `message_consume_log` | `message-service` | 消费幂等和消费结果追踪 |
| `api_access_log`、`operation_log` | `gateway` / `ops-service` | 网关访问日志和后台操作审计 |
| `decoction_task`、`decoction_task_event`、`decoction_device_work_record` | `decoction-service` | 煎煮任务、事件和设备作业记录 |
| `device_*` | `device-service` | 设备档案和占用关系 |
| `shipment`、`shipment_trace` | `logistics-service` | 物流单、轨迹、签收 |
| `callback_record` | `callback-service` | 回调任务、响应、失败补偿 |
| `portal_address_supplement` | `portal-service` | 地址补录申请，不直接修改订单地址 |
| `integration_message`、`integration_retry_task` | `integration-service` | 外围消息归档和真实回推重试任务 |

数据规则：

- 主业务写入和 Outbox 写入必须在同一事务中完成。
- 状态字段必须由状态机或应用服务统一推进，不能散落 SQL 随意更新。
- 外部原始报文要保留 raw payload 或扩展字段，内部领域字段保持稳定。
- 过渡表结构可以暂由一个服务的 Flyway 创建，但不能改变逻辑归属。

## 6. 事件边界

事件只表达已经发生的业务事实，不承担复杂业务判断。

| 事件 | 来源 | Topic | 主要消费方 | 用途 |
| --- | --- | --- | --- | --- |
| `ORDER_CREATED` | `order-service` | `zhyf-order-event` | `message-service`、`workflow-service` | 订单创建后校验、生成审核任务、后续回调准备 |
| `ORDER_REVIEWED` | `workflow-service` | `zhyf-order-event` | `callback-service`、`ops-service` | 审核结果通知、审计、后续履约 |
| `PRESCRIPTION_DISPENSED` | `workflow-service` | `zhyf-prescription-event` | `decoction-service`、`callback-service` | 调剂完成后进入复核/煎煮准备 |
| `PRESCRIPTION_RECHECKED` | `workflow-service` | `zhyf-prescription-event` | `decoction-service`、`device-service` | 复核完成后进入煎煮/设备作业 |
| `DECOCTION_STARTED` | `decoction-service` | `zhyf-decoction-event` | `callback-service`、`ops-service` | 煎煮开始通知和追踪 |
| `DECOCTION_FINISHED` | `decoction-service` | `zhyf-decoction-event` | `logistics-service`、`callback-service` | 煎煮完成后打包物流 |
| `LOGISTICS_SHIPPED` | `logistics-service` | `zhyf-logistics-event` | `callback-service`、`ops-service` | 发货回调 |
| `ORDER_CALLBACK_FAILED` | `callback-service` | `zhyf-callback-event` | `ops-service` | 回调失败告警和人工补偿 |

事件规则：

- 所有业务事件先写 `event_outbox`，再由 `message-service` 投递 RocketMQ。
- 消费端必须有幂等记录，重复消费不能重复改业务状态。
- 消费失败必须记录原因，后续进入重试、死信或人工处理。
- payload 必须带 `tenantId`、业务主键、业务编号和可扩展字段。

## 7. 核心业务链路

### 7.1 机构推单到审核

```text
机构推单
 -> gateway 签名/IP/访问日志
 -> order-service 创建订单、处方基础数据、状态日志
 -> order-service 写 event_outbox: ORDER_CREATED
 -> message-service 发布 RocketMQ
 -> message-service 消费 ORDER_CREATED 并写 order_validation_record
 -> workflow-service 消费 ORDER_CREATED
 -> workflow-service 根据校验结果创建 workflow_task: ORDER_REVIEW/PENDING
 -> admin-web 查询审核任务
 -> workflow-service 审核通过/拒绝
 -> order-service 内部接口推进 order_main.status
 -> order_status_log 记录状态变化
```

### 7.2 审核到调剂复核

```text
审核通过
 -> workflow-service 生成 PRESCRIPTION_DISPENSE/PENDING
 -> admin-web 查询调剂任务
 -> workflow-service 完成调剂并写 dispense_record
 -> workflow-service 生成 PRESCRIPTION_RECHECK/PENDING
 -> admin-web 查询复核任务
 -> workflow-service 完成复核
 -> order-service 内部接口推进订单到 RECHECKED
 -> order_status_log 记录复核状态变化
```

### 7.3 煎煮到物流回调

```text
PDA/MES 上报煎煮状态
 -> decoction-service 记录煎煮任务、事件和设备作业明细
 -> 订单推进到 DECOCTING / DECOCTED
 -> logistics-service 查询 DECOCTED 待打包订单
 -> logistics-service 打包并推进订单到 PACKED
 -> logistics-service 发货/签收或接收顺丰/EMS 轨迹回调
 -> callback-service 创建机构回调记录
 -> callback-service 真实 HTTP 派发，失败后排期重试，超过上限进入 DEAD
 -> ops-service 汇总异常和补偿入口
```

### 7.4 门户、报表和运维

```text
portal-service 提供医院查单和地址补录申请
report-service 提供只读统计和 CSV 导出
ops-service 聚合 Outbox、消费日志、校验记录、访问日志、回调失败、集成失败和核心健康概览
admin-web 统一承接后台操作、查询、排错和导出入口
```

## 8. 前端后台平台规划

`admin-web` 要恢复旧平台后台的核心业务功能，而不是只做开发调试页。当前后端核心能力已逐步补齐，下一步前端优先按旧平台后台工作台还原这些区域：

- 工作台：首页核心待处理量、异常量、最近订单、健康概览。
- 订单中心：订单列表、筛选、详情、履约进度、状态日志、人工操作入口。
- 处方中心：处方列表、处方详情、处方明细、调剂/复核相关信息。
- 药房作业：审核任务、调剂任务、复核任务、任务处理和结果回看。
- 煎煮作业：PDA/MES 模拟入口、可操作处方、设备、任务事件、作业明细。
- 物流回调：待打包、物流单、轨迹、签收、回调记录、人工补偿。
- 门户与集成：医院查单、地址补录、集成消息、重试任务。
- 报表统计：核心概览、趋势、状态分布、CSV 导出。
- 运维排错：Outbox、消费日志、校验记录、访问日志、回调失败、集成失败、健康概览。

前端规则：

- 管理端不直接连数据库，只调用服务 API。
- 页面必须围绕真实业务动作和后台操作效率设计，不做空壳展示。
- 本地开发通过 Vite proxy 或 gateway 统一转发。
- 机构密钥、管理端账号密码等敏感值不写入前端仓库。

## 9. 包结构规则

业务服务默认四层：

```text
com.zhyf.<service>
├── api
├── application
├── domain
├── infrastructure
└── <Service>Application
```

技术型服务可按技术职责拆：

```text
com.zhyf.message
├── consumer
├── publisher
├── job
├── application
├── infrastructure
└── MessageServiceApplication
```

规则：

- `api` 只做协议适配和参数转换。
- `application` 管事务边界、用例编排、跨服务调用。
- `domain` 放状态机、业务规则、值对象。
- `infrastructure` 放 JDBC、RocketMQ、HTTP client、配置、外部系统 adapter。
- 不把 Controller、SQL、业务判断混在同一个类里。
- 不在 `zhyf-common` 放具体业务规则。

## 10. 分阶段实施路线

### 阶段一：核心接入、订单和审核

目标：

- 建立后端多模块工程和本地联调环境。
- 跑通机构推单、订单落库、Outbox、RocketMQ、校验、审核任务、审核状态推进。
- 提供最小 admin-web 管理入口。

当前状态：后端真实联调已跑通，`admin-web` 已有第一版。

### 阶段二：履约、调剂复核和运维查询

目标：

- 补齐调剂、复核任务模型和状态机。
- 补齐订单履约进度、调剂记录、ops 查询入口和后台排错入口。

当前状态：审核、调剂、复核闭环已落地；订单履约进度、核心健康概览、报表 CSV 导出已补齐。

### 阶段三：PDA/MES、煎煮、物流和回调补偿

目标：

- 用模拟器替代真实 PDA、MES、物流和外围系统完成完整履约链路。
- 把设备、煎煮、物流、回调失败补偿从核心订单链路中隔离。
- 后续接真实设备和外部系统时只替换 adapter，不改订单主流程。

当前状态：PDA/MES 模拟器、旧协议兼容入口、物流最小链路、真实 HTTP 回调发送、失败重试、`DEAD` 终态和运维聚合已落地。

### 阶段四：门户、报表、集成、性能和后台还原

目标：

- 补齐医院查单、地址补录、报表、统计和导出。
- 做核心接口压测、容量评估、监控告警和稳定性优化。
- 恢复旧平台后台核心功能体验，完成前后端真实联调。
- 主链路稳定后再接入低风险 AI 提效能力。

当前状态：`portal-service`、`report-service`、`integration-service` 已有最小能力；当前下一优先级是 `admin-web` 按旧平台后台还原核心页面和真实端到端联调。社康完整拉单暂缓。

## 11. 当前落地状态

当前最小核心链路已经推进到：

```text
机构推单
 -> gateway
 -> order-service
 -> event_outbox
 -> message-service
 -> workflow-service 审核
 -> workflow-service 调剂
 -> workflow-service 复核
 -> decoction-service 煎煮
 -> logistics-service 物流
 -> callback-service 回调补偿
 -> portal/report/integration/ops/admin-web 基础入口
```

后端核心业务能力已覆盖订单创建、审核、调剂、复核、煎煮、物流、回调、门户查单、报表导出、集成重试和运维排错。主要缺口在：

- `admin-web` 还没有完全还原旧平台后台的核心业务操作体验。
- 核心链路还需要一次前后端端到端联调验收。
- 社康配置化拉单暂缓，后续在核心闭环稳定后单独推进。
- 性能压测、监控告警和生产部署清单还需要按部署环境收口。

## 12. 技术取舍

- 当前后端使用 JDBC，表结构还在快速确认阶段，SQL 可控，便于真实联调。
- 暂不引入完整 ORM；等聚合关系、查询复杂度稳定后再评估 MyBatis 或 JPA。
- 暂不引入复杂流程引擎；workflow 先用任务表和状态机承载。
- 暂不接真实 PDA/MES 设备；先用模拟器验证协议、幂等和状态推进。
- 当前开发阶段允许配置开发默认密码和本地代理，但生产部署必须改为外部密钥管理或环境变量注入。

## 13. 禁止事项

- 不用“先快点做出来”为理由长期混用服务职责。
- 不把外部协议、业务规则、DB 访问、消息处理堆在一个类或一个包里。
- 不绕过状态机直接随意更新 `order_main.status`。
- 不让前端直接访问数据库。
- 不清库、清日志、删 volume，除非用户明确要求。
- 不部署到生产或共享服务器，除非用户明确要求。
- 不在文档或代码中保存生产密钥、私钥、生产数据库密码。
