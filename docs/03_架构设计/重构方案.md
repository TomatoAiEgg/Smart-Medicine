# 智能药房 SaaS 平台重构方案

本文档是基于 `架构总蓝图` 展开的目标架构说明。若与 `docs/03_架构设计/架构总蓝图.md` 冲突，以蓝图为准。

## 1. 项目定位

本项目重构目标是把旧系统中分散在平台后台、对外 API、异步任务、社康集成、医院查单和地址补录中的能力，统一收敛为一个可长期演进的多租户 SaaS 平台。

第一版重构不是做一个“更小的版本”，而是做一个可以替代旧系统生产闭环的版本。

## 2. 重构原则

- 先替代旧系统核心业务，再做增强能力。
- 外部协议优先兼容，内部实现按领域重构。
- 订单、处方、煎煮、物流、回调都要可审计、可追踪、可回放。
- 人工修正与正常履约流程分离。
- 消息发送、第三方回调、社康回写必须有幂等和重试。
- AI 能力不进入第一版主链路。

## 4. 总体架构

```text
Browser / PDA 模拟器 / MES 模拟器 / 社康集成 / 医院门户 / 地址补录入口 / 外部机构
        |
Gateway
        |
        +-- auth-institution
        +-- order-service
        +-- prescription-service
        +-- workflow-service
        +-- decoction-service
        +-- device-service
        +-- logistics-service
        +-- callback-service
        +-- integration-service
        +-- portal-service
        +-- message-service
        +-- ops-service
        +-- report-service
        |
PostgreSQL + Redis + RocketMQ
```

## 6. 关键设计点

### 6.1 状态机优先

订单、处方、煎煮、物流都不能再直接靠多处 `update` 拼状态。

必须通过状态机统一推进，状态变更同时写日志和事件。

### 6.2 事件驱动

写业务表和写事件表必须在同一事务内完成。

事件发布后由独立消费者完成：

- 审方
- 短信
- 回调
- 社康回写
- 物流通知
- 报表刷新

### 6.3 适配器模式

医院、社康、物流、MES、地址补录都应该通过适配器接入。

适配器只做协议转换，不承载核心业务判断。

### 6.4 审计优先

以下操作必须可审计：

- 手工审核
- 调剂
- 复核
- 取消
- 初始化
- 手工签收
- 订单走流程
- 地址修改
- 社康状态回写重试

## 8. 第一版最小生产闭环

第一版必须跑通：

```text
机构推单
 -> 订单创建
 -> 审核
 -> 调剂
 -> 复核
 -> PDA 绑定
 -> MES 模拟煎煮
 -> 打包发货
 -> 物流回调
 -> 机构回调
 -> 医院查单
```

## 9. 当前结论

重构方案的核心是“领域拆分 + 事件驱动 + 协议兼容 + 状态机治理”。

如果这四件事先立住，后续迁移旧功能、做 AI 提效、做运营扩展都会轻很多。
