# 总体架构落地设计

本文档把 `架构总蓝图` 落到可执行的服务边界、数据边界、事件边界、前端入口、包结构和实施路线。本文不是单独的“第一阶段架构”，而是总架构的落地方案；第一阶段、第二阶段、第三阶段只表示实施顺序和交付范围。

## 1. 总体目标

新系统目标是用一套多租户 SaaS 架构替代旧智慧药房后台、机构接口、医院查单、社康集成、PDA/MES 作业、物流回调和运维补偿能力。

总体架构必须同时满足：

- 外部机构接入可兼容旧接口并逐步规范化。
- 订单、处方、审核、调剂、复核、煎煮、物流、回调职责边界清晰。
- 核心写入链路使用事务内 Outbox，事件投递和消费可追踪、可补偿。
- PDA、MES、物流、社康、医院查单等外部系统通过 adapter 或独立服务隔离。
- 管理端、运维端、模拟器都基于同一套后端能力，不重复造业务逻辑。
- 分阶段交付时允许过渡承载，但过渡必须写明逻辑归属和迁移目标。

## 2. 目标服务边界

| 服务 | 目标职责 | 当前落地说明 |
| --- | --- | --- |
| `gateway` | 统一入口、路由、机构签名校验、IP 白名单、限流、访问日志 | 已承接机构推单入口转发和访问日志 |
| `auth-institution` | 租户、机构、应用密钥、白名单、回调配置、后台用户权限基础 | 已承接机构应用内部查询 |
| `order-service` | 订单创建、订单查询、订单状态机、订单状态日志、人工修正入口 | 已承接机构推单、订单/处方基础落库、Outbox、内部状态推进 |
| `prescription-service` | 处方、处方明细、处方修改、审方摘要、处方状态机 | 当前只有模块骨架，处方表暂由 `order-service` 承载 |
| `workflow-service` | 后台审核、调剂、复核、订单锁、人工流程控制、工作任务、状态推进 | 已承接订单审核任务、复核任务和状态推进 |
| `message-service` | Outbox 扫描发布、RocketMQ 生产、消费入口、消费幂等、消息补偿 | 已承接 `ORDER_CREATED` 发布和基础校验记录 |
| `decoction-service` | PDA 作业、扫码绑定、水桶/设备、加水、煎煮、MES 适配 | 已落地 PDA/MES 模拟器、设备作业明细、旧协议兼容入口和 adapter 抽象 |
| `device-service` | 设备档案、设备占用、设备关系、设备状态 | 待落地，可与煎煮域分阶段拆分 |
| `logistics-service` | 打包出库、面单、物流轨迹、签收、自提、顺丰/EMS 适配 | 已落地最小链路：待打包订单、打包、发货、签收、轨迹记录和顺丰/EMS 回调兼容入口 |
| `callback-service` | 机构状态回调、社康回写、地址补录回推、失败重试、死信、人工重放 | 已落地回调记录和人工补偿骨架，真实 HTTP 发送和重试调度待补 |
| `integration-service` | 社康、医院查单、地址补录、旧系统外围协议适配 | 待落地 |
| `portal-service` | 医院查单、地址补录等外部轻量门户 | 待落地 |
| `ops-service` | 操作日志、异常日志、审计、链路追踪、监控聚合、联调排错 | 已有最小只读查询接口，已接入 `admin-web` 运维排错页 |
| `report-service` | 报表、统计、导出、经营分析 | 后续增强 |
| `zhyf-common` | 通用响应、异常、状态枚举、签名、审计字段、少量基础工具 | 已落地 |
| `admin-web` | 管理端订单、处方、审核、调剂、复核、设备、物流、回调、运维页面 | 已有订单查询、审核/复核任务、煎煮模拟、物流回调和运维排错入口 |

服务拆分原则：

- 服务边界按业务归属决定，不按当前代码方便程度决定。
- 当前可以临时由已落地服务承载表结构或接口，但逻辑归属必须以本表为准。
- 跨服务状态推进优先通过明确的应用服务接口或事件，不允许直接跨服务改库。

## 3. 数据边界

| 数据 | 逻辑归属 | 落地策略 |
| --- | --- | --- |
| `tenant` | `auth-institution` | 租户基础表，当前可由已有 migration 创建，后续归属认证机构域 |
| `institution` | `auth-institution` | 机构基础表，后续由机构域维护 |
| `institution_app` | `auth-institution` | 应用密钥、回调地址、启停状态 |
| `institution_ip_whitelist` | `auth-institution` | 机构 IP 白名单 |
| `order_main` | `order-service` | 订单主表 |
| `order_status_log` | `order-service` | 订单状态日志，所有状态推进必须写入 |
| `prescription` | `prescription-service` | 当前由 `order-service` migration 创建，后续迁移到处方域维护 |
| `prescription_detail` | `prescription-service` | 当前由 `order-service` migration 创建 |
| `workflow_task` | `workflow-service` | 工作任务表，已由 `workflow-service` 逻辑接管 |
| `order_validation_record` | `message-service` / `workflow-service` 读取 | 订单事件基础校验结果，供 workflow 创建任务前判断 |
| `event_outbox` | 平台事件基础设施 | 业务服务写入，`message-service` 发布 |
| `message_consume_log` | `message-service` | 消费幂等和消费结果追踪 |
| `api_access_log` | `gateway` / `ops-service` | 网关访问日志和运维查询 |
| `operation_log` | `ops-service` | 后台操作审计 |
| `decoction_task` | `decoction-service` | 煎煮任务、设备作业、PDA 状态 |
| `device_*` | `device-service` | 设备档案和设备占用 |
| `shipment`、`shipment_trace` | `logistics-service` | 物流单、轨迹、签收 |
| `callback_record` | `callback-service` | 回调任务、回调日志、失败补偿 |
| `integration_*` | `integration-service` | 社康、医院、地址补录协议映射 |

数据规则：

- 主业务写入和 Outbox 写入必须在同一事务中完成。
- 状态字段必须由状态机或应用服务统一推进，不能散落 SQL 随意更新。
- 过渡表结构可以暂由一个服务的 Flyway 创建，但不能改变逻辑归属。
- 外部原始报文要保留 raw payload 或扩展字段，内部领域字段保持稳定。

## 4. 事件边界

事件只表达已经发生的业务事实，不承载复杂业务判断。

| 事件 | 来源 | Topic | 主要消费方 | 用途 |
| --- | --- | --- | --- | --- |
| `ORDER_CREATED` | `order-service` | `zhyf-order-event` | `message-service`、`workflow-service` | 订单创建后校验、生成审核任务、后续回调准备 |
| `ORDER_REVIEWED` | `workflow-service` | `zhyf-order-event` | `callback-service`、`ops-service` | 审核结果通知、审计、后续履约 |
| `PRESCRIPTION_DISPENSED` | `workflow-service` | `zhyf-prescription-event` | `decoction-service`、`callback-service` | 调剂完成后进入复核/煎煮准备 |
| `PRESCRIPTION_RECHECKED` | `workflow-service` | `zhyf-prescription-event` | `decoction-service`、`device-service` | 复核完成后进入煎煮/设备作业 |
| `DECOCTION_STARTED` | `decoction-service` | `zhyf-decoction-event` | `callback-service`、`ops-service` | 煎煮开始通知和追踪 |
| `DECOCTION_FINISHED` | `decoction-service` | `zhyf-decoction-event` | `logistics-service`、`callback-service` | 煎煮完成后打包物流 |
| `LOGISTICS_SHIPPED` | `logistics-service` | `zhyf-logistics-event` | `callback-service`、`ops-service` | 发货回调 |
| `ORDER_CALLBACK_FAILED` | `callback-service` | `zhyf-callback-event` | `ops-service` | 回调失败告警和人工补偿 |

事件规则：

- 所有业务事件先写 `event_outbox`，再由 `message-service` 投递 RocketMQ。
- 消费端必须有幂等记录，重复消费不能重复改业务状态。
- 消费失败必须记录原因，后续进入重试、死信或人工处理。
- 事件 payload 必须带 `tenantId`、业务主键、业务编号和可扩展字段。

## 5. 核心链路

### 5.1 机构推单到审核

```text
机构推单
 -> gateway 签名/IP/访问日志
 -> order-service 创建订单、处方基础数据、状态日志
 -> order-service 写 event_outbox: ORDER_CREATED
 -> message-service 发布 RocketMQ
 -> message-service 消费 ORDER_CREATED 并写 order_validation_record
 -> workflow-service 消费 ORDER_CREATED
 -> workflow-service 根据校验结果创建 workflow_task: ORDER_REVIEW/PENDING
 -> admin-web 查询审核任务
 -> workflow-service 审核通过/拒绝
 -> order-service 内部接口推进 order_main.status
 -> order_status_log 记录状态变化
```

### 5.2 审核到调剂复核

```text
审核通过
 -> workflow-service 生成调剂任务
 -> 调剂完成后推进处方/订单履约状态
 -> workflow-service 生成复核任务
 -> 复核完成后发布 PRESCRIPTION_RECHECKED
 -> decoction-service 接收进入煎煮准备
```

### 5.3 煎煮到物流回调

```text
PDA/MES 上报煎煮状态
 -> decoction-service 记录煎煮任务和设备状态
 -> 煎煮完成发布 DECOCTION_FINISHED
 -> logistics-service 查询 DECOCTED 待打包订单
 -> logistics-service 打包并推进订单到 PACKED
 -> logistics-service 发货/签收或接收顺丰/EMS 轨迹回调，写 shipment_trace
 -> callback-service 创建机构回调记录
 -> admin-web 人工查看、标记失败/成功或重放回调记录
 -> ops-service 汇总异常和补偿入口
```

## 6. 包结构规则

业务服务默认四层：

```text
com.zhyf.<service>
├── api
├── application
├── domain
├── infrastructure
└── <Service>Application
```

技术型服务可按技术职责拆：

```text
com.zhyf.message
├── consumer
├── publisher
├── job
├── application
├── infrastructure
└── MessageServiceApplication
```

包结构规则：

- `api` 只做协议适配和参数转换。
- `application` 管事务边界、用例编排、跨服务调用。
- `domain` 放状态机、业务规则、值对象。
- `infrastructure` 放 JDBC、RocketMQ、HTTP client、配置、外部系统 adapter。
- 不把 Controller、SQL、业务判断混在同一个类里。
- 不在 `zhyf-common` 放具体业务规则。

## 7. 前端入口

| 前端 | 目标页面 | 当前落地 |
| --- | --- | --- |
| `admin-web` | 订单中心、处方中心、审核/调剂/复核、PDA/MES 作业监控、物流中心、回调补偿、机构配置、设备管理、系统监控 | 已有订单查询、审核/复核任务处理、煎煮模拟、物流回调和运维排错入口 |
| `pda-simulator-web` | PDA 登录、扫码绑定、加水、煎煮状态上报、重复/并发/超时模拟 | 待落地 |
| `integration-simulator` | 机构推单、医院查单、社康拉单、物流轨迹、地址补录、回调失败模拟 | 待落地 |

前端规则：

- 管理端不直接连数据库，只调用服务 API。
- 本地开发通过 Vite proxy 或 gateway 统一转发。
- 机构密钥、管理端账号密码等敏感值不写入前端仓库。
- 模拟器用于替代暂无真实硬件/外部系统的联调，不进入核心业务服务。

## 8. 分阶段实施路线

阶段是交付节奏，不是架构分叉。每一阶段都必须对齐本文的目标边界。

### 阶段一：核心接入与订单审核闭环

目标：

- 建立可持续演进的后端多模块工程和本地联调环境。
- 跑通机构推单、订单落库、Outbox、RocketMQ、校验、审核任务、审核状态推进。
- 提供最小 admin-web 管理入口。

范围：

- `gateway`
- `auth-institution`
- `order-service`
- `message-service`
- `workflow-service`
- `admin-web`
- `zhyf-common`

验收：

- `mvn test`、`mvn -DskipTests package` 通过。
- 机构推单能创建订单、处方、处方明细、状态日志和 Outbox。
- `event_outbox.NEW` 能发布到 RocketMQ 并更新为 `SENT`。
- `ORDER_CREATED` 消费能写入 `message_consume_log` 和 `order_validation_record`。
- 校验通过能创建 `workflow_task: ORDER_REVIEW/PENDING`。
- 管理端能查询订单和处理审核任务。
- 审核通过/拒绝能推进订单状态并写状态日志。

当前状态：阶段一核心链路已完成后端真实联调，`admin-web` 已有第一版。

### 阶段二：处方履约、调剂复核与运维查询

目标：

- 把处方能力从订单创建的临时承载中逐步拆清楚。
- 补齐调剂、复核的最小任务模型和状态机。
- 补齐 ops 查询入口，支持联调和异常定位。

范围：

- `prescription-service`
- `workflow-service` 调剂/复核能力
- `ops-service` 最小查询接口
- `admin-web` 处方、调剂、复核、运维页面

验收：

- 处方状态能从创建推进到调剂、复核相关状态。
- 调剂/复核操作全部写状态日志和操作记录。
- `ops-service` 能查询 Outbox、消费日志、校验记录、访问日志。
- 管理端能通过运维排错页查看 Outbox、消费日志、订单校验和访问日志。
- 管理端能看到订单/处方/任务的完整履约进度。

### 阶段三：PDA/MES、煎煮、物流与回调补偿

目标：

- 用模拟器先替代真实 PDA、MES、物流和外围系统完成完整履约链路。
- 把设备、煎煮、物流、回调失败补偿从核心订单链路中隔离。
- 后续接真实设备和外部系统时只替换 adapter，不改订单主流程。

范围：

- `decoction-service`
- `device-service`
- `logistics-service`
- `callback-service`
- `integration-service`
- `pda-simulator-web`
- `integration-simulator`
- `admin-web` 设备、煎煮、物流、回调页面

验收：

- PDA/MES 模拟器能推进煎煮任务状态。
- 旧 PDA/MES 兼容入口能映射到同一套煎煮业务状态机。
- 煎煮完成后能触发物流或自提流程。
- 回调失败能进入重试、死信或人工重放。
- ops/admin-web 能查询端到端链路和失败原因。

当前状态：PDA/MES 模拟器、旧协议兼容入口、物流最小链路、回调记录和 `admin-web` 物流回调页已落地；真实外部 HTTP 回调发送、自动重试、死信告警和完整本地联调待补。

### 阶段四：门户、报表、性能与 AI 提效

目标：

- 补齐医院查单、地址补录、报表、统计和导出。
- 做压测、容量评估、监控告警和稳定性优化。
- 在主链路稳定后接入低风险 AI 提效能力。

范围：

- `portal-service`
- `report-service`
- `ops-service` 监控聚合
- `admin-web` 报表和系统监控
- AI 文档摘要、联调日志摘要、规则辅助等非主链路能力

验收：

- 关键接口有压测结果和容量基线。
- 监控、日志、告警能覆盖核心服务。
- 报表和门户能力不影响订单主链路稳定性。

## 9. 当前技术取舍

- 当前后端使用 JDBC，原因是表结构还在快速确认阶段，SQL 可控，便于真实联调。
- 暂不引入完整 ORM；等聚合关系、查询复杂度稳定后再评估 MyBatis 或 JPA。
- 暂不强行引入复杂流程引擎；workflow 先用任务表和状态机承载。
- 暂不接真实 PDA/MES 设备；先用模拟器验证协议、幂等和状态推进。
- 当前开发阶段允许配置开发默认密码和本地代理，但生产部署必须改为外部密钥管理或环境变量注入。

## 10. 禁止事项

- 不用“先快点做出来”为理由长期混用服务职责。
- 不把外部协议、业务规则、DB 访问、消息处理堆在一个类或一个包里。
- 不绕过状态机直接随意更新 `order_main.status`。
- 不让前端直接访问数据库。
- 不在文档或代码中保存生产密钥、私钥或生产数据库密码。
