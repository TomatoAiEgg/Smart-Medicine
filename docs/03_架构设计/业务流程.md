# 业务流程与状态机设计

本文档是 `架构总蓝图` 的流程展开版。流程和状态机设计必须服从 `docs/03_架构设计/架构总蓝图.md`。

## 1. 总体流程

```text
机构推单
 -> 订单创建
 -> 异步校验/审方/药品匹配
 -> 后台审核
 -> 调剂打单
 -> 复核
 -> 水桶绑定
 -> PDA/设备绑定煎煮机
 -> 开始煎煮
 -> 煎煮完成
 -> 打包/面单/自提
 -> 物流发货
 -> 物流签收
 -> 机构状态回调
 -> 医院查单/轨迹查询
```

## 2. 机构推单流程

1. 外部机构调用兼容接口。
2. Gateway 校验 IP、时间戳、签名、接口权限。
3. `order-service` 做幂等校验。
4. 检查机构处方号是否重复。
5. 写 `order_main`。
6. 写 `prescription`。
7. 写 `prescription_detail`。
8. 写接单履约轨迹。
9. 写 `event_outbox: OrderCreated`。
10. 返回旧协议响应。

失败处理：

- 验签失败：直接返回失败，不落订单。
- 重复推单：返回重复信息。
- 部分落库失败：事务回滚。
- 事件发送失败：Outbox 保证后续发布。

## 3. 创建后异步处理流程

事件：`OrderCreated`

消费者依次处理：

1. 药品目录匹配。
2. 订单基础校验。
3. 处方校验。
4. 地址/位置校验。
5. 楼层建议。
6. 煎药机方案建议。
7. 审方平台调用。
8. 写校验结果和审方结果。

异常处理：

- 单个步骤失败不应导致订单丢失。
- 失败写异常日志和校验结果。
- 可恢复失败进入重试。

## 4. 后台审核流程

```text
待审核订单
 -> 锁单
 -> 查看订单和处方明细
 -> 修改处方/明细
 -> 保存审核
 -> 订单状态 5 -> 15
 -> 写审核记录
 -> 写状态日志
 -> 写履约轨迹
 -> 发布 OrderReviewed
 -> 回调机构
```

规则：

- 只有状态 `5` 或等价待审核状态可审核。
- 审核保存必须记录操作人。
- 审核修改处方明细需要保留修改痕迹。

## 5. 调剂流程

```text
订单状态 15
 -> 调剂列表
 -> 输入调剂工号
 -> 校验调剂权限
 -> 写调剂人和调剂时间
 -> 更新打印状态
 -> 返回打印数据
 -> 写履约轨迹
```

规则：

- 调剂不直接把订单推进到 `20`。
- 调剂记录按处方维度保存。

## 6. 复核流程

```text
订单状态 15
 -> 扫处方号
 -> 校验处方可复核
 -> 校验复核工号
 -> 校验重量和地址
 -> 代煎处方绑定水桶
 -> 写复核人和复核时间
 -> 全部处方复核完成
 -> 订单状态 15 -> 20
 -> 发布 PrescriptionReviewed / OrderRechecked
 -> 回调机构
```

规则：

- 处方已取消不能复核。
- 订单状态小于 `15` 不能复核。
- 订单状态大于等于 `20` 不能重复复核。
- 代煎处方必须有加水量和水桶。
- 同一状态回调必须幂等，避免重复 `20` 回调。

## 7. PDA/设备煎煮流程

### 7.1 绑定设备

```text
复核完成
 -> PDA 查询可操作处方
 -> 扫描处方号
 -> 扫描煎煮机
 -> 校验设备空闲
 -> 写绑定记录
 -> 设备占用
```

### 7.2 开始煎煮

```text
设备已绑定
 -> PDA/设备上报开始
 -> 绑定状态 0 -> 1
 -> 处方 boil_status 0 -> 1
 -> 订单状态 20/23 -> 25
 -> 写煎煮开始时间
 -> 写履约轨迹
 -> 发布 DecoctionStarted
```

### 7.3 完成煎煮

```text
煎煮中
 -> PDA/设备上报完成
 -> 绑定状态 1 -> 2
 -> 处方 boil_status 1 -> 2
 -> 订单状态 25 -> 27
 -> 写煎煮结束时间
 -> 释放设备
 -> 写履约轨迹
 -> 发布 DecoctionFinished
 -> 回调机构
```

### 7.4 取消/终止

- `operStatus=8`：取消煎煮。
- `operStatus=9`：终止煎煮。
- 都必须释放设备。
- 都必须写操作日志。

## 8. 打包与物流流程

### 8.1 面单和打包

```text
订单状态 >= 20
 -> 进入面单页面
 -> 查询物流费
 -> 创建物流单
 -> 写运单号
 -> 写包裹重量/包裹数
 -> 订单状态 -> 30
 -> 写打包时间
 -> 发布 OrderPacked
 -> 回调机构
```

### 8.2 自提

```text
订单状态 >= 20
 -> 自提完成
 -> 写包裹信息
 -> 写出库时间
 -> 写签收时间
 -> 订单状态 -> 40
 -> 发布 OrderSigned
 -> 回调机构
```

### 8.3 物流回调

顺丰：

```text
50/43/54 -> 36
80/8000  -> 40
其他且已发货 -> 37
```

EMS：

```text
203     -> 36
748/704 -> 40
其他且已发货 -> 37
```

规则：

- 物流状态不能倒退。
- 合并单回调需要更新原订单。
- 原始回调报文必须保存。

## 9. 机构状态回调流程

```text
订单状态变化
 -> 发布 OrderStatusChanged
 -> callback-service 判断是否需要回调
 -> 生成 callback_record
 -> 签名
 -> 调用机构回调 URL
 -> 成功记录
 -> 失败按策略重试
 -> 超过次数进入死信
```

第一版需要兼容旧系统回调状态：

- `15`
- `20`
- `27`
- `30`
- `40`
- `98`
- `99`

社康回调特殊规则：

- 部分社康只需要退款/取消类回调。
- 规则必须配置化。

## 10. 社康拉单流程

```text
定时任务按区域拉消息
 -> 保存 ch_message 等价消息
 -> 获取社康订单详情
 -> 转换为内部创建订单请求
 -> 创建订单
 -> 标记本地消息已处理
 -> 回写社康状态
 -> 回写失败写 retry_task
 -> 定时重试
```

规则：

- 远端消息获取后必须先本地持久化。
- 上传成功但回写失败不能重复创建订单。
- 社康机构映射必须配置化。

## 11. 医院查单流程

```text
医院用户登录
 -> 获取可见机构范围
 -> 查询处方维度列表
 -> 查看订单详情
 -> 查看物流/履约轨迹
 -> 导出
```

规则：

- `companyNums=all` 可看全部机构。
- 非 `all` 只能看授权机构。
- 查询返回需要脱敏策略。

## 12. 地址补录流程

```text
患者打开地址补录入口
 -> 按 companyNum 查询配置
 -> 跳转医院页面
 -> 提交地址
 -> Redis 防重复提交
 -> 保存/更新地址
 -> 按医院适配器回推
 -> 写推送日志
```

规则：

- 地址保存和回推日志必须都落库。
- 回推失败要支持人工补推或重试。
- 医院协议差异不能写死在核心服务。

## 13. 人工修正流程

### 13.1 初始化处方

```text
人工选择初始化
 -> 订单回 5
 -> 处方状态重置
 -> 绑定记录失效
 -> 释放设备
 -> 如已打包则取消物流
 -> 写人工修正记录
```

### 13.2 取消处方

```text
人工取消处方
 -> 写处方取消结果
 -> 判断是否整单取消
 -> 整单取消时订单到 99
 -> 回调取消状态
 -> 写人工修正记录
```

### 13.3 订单走流程

```text
人工补齐审核/调剂/复核/煎煮/签收时间
 -> 订单直接到 40
 -> 写订单备注
 -> 写履约轨迹
 -> 回调机构
 -> 写人工修正记录
```

## 14. 状态机设计

### 14.1 订单状态机

| 当前状态 | 动作 | 目标状态 |
| --- | --- | --- |
| `5` | 审核通过 | `15` |
| `15` | 全部处方复核完成 | `20` |
| `20` | 开始煎煮 | `25` |
| `25` | 煎煮完成 | `27` |
| `20/27` | 打包完成 | `30` |
| `30` | 物流发货 | `36` |
| `36` | 物流运输中 | `37` |
| `36/37` | 签收 | `40` |
| `5/15` | 机构取消 | `99` |
| `5/15` | 审核失败 | `98` |
| 任意非终态 | 人工签收/走流程 | `40` |
| 任意非终态 | 人工初始化 | `5` |

禁止：

- 终态回到普通状态，除非走人工修正。
- 物流状态倒退。
- 已复核重复发 `20` 回调。

### 14.2 处方作业状态

```text
WAIT_AUDIT
 -> AUDIT_PASSED
 -> DISPENSED
 -> RECHECKED
 -> DECOCTION_BOUND
 -> DECOCTING
 -> DECOCTED
 -> FINISHED
```

取消状态：

- `CANCELLED`
- `AUDIT_FAILED`

### 14.3 煎煮任务状态

```text
WAIT_BIND
 -> BOUND
 -> WATER_BOUND
 -> DECOCTING
 -> DECOCTED
```

异常状态：

- `CANCELLED`
- `TERMINATED`

### 14.4 物流状态

```text
WAIT_PACK
 -> PACKED
 -> SHIPPED
 -> IN_TRANSIT
 -> SIGNED
```

异常状态：

- `CANCELLED`
- `EXCEPTION`

## 15. 状态机实现要求

- 状态变更必须经过状态机服务。
- 状态变更必须写 `order_status_log`。
- 状态变更必须写业务事件。
- 状态变更必须支持幂等键。
- 非法状态跳转返回明确错误。
- 人工修正必须走单独入口并记录原因。

## 16. 当前结论

旧系统最大风险是状态分散更新。

新系统第一版必须优先把状态机收敛好，否则后续消息、回调、PDA、物流都会继续混乱。
