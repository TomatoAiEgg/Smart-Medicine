# admin-web 前端重构方案

本文档是 `admin-web` 的专项前端架构与重构方案。总架构仍以 `docs/03_架构设计/重构方案.md` 为准，本文只展开后台管理端如何从当前联调页合集重构为可替代旧平台后台的业务工作台。

## 1. 当前问题

重构前 `frontend/admin-web` 已经接入订单查询、审核、调剂、复核、煎煮、物流、门户查单、报表、集成和运维接口，但实现形态仍偏开发联调页：

- `src/App.vue` 约 3000 行，集中承载导航、状态、请求、表格、表单和所有业务视图。
- 当前一级菜单以接口联调视角组织，例如“煎煮模拟”“物流回调”“集成适配”“运维排错”，不是旧平台后台的日常业务入口。
- 订单、处方、任务、物流、回调、报表之间缺少统一详情页、抽屉和跳转关系。
- 表格筛选、分页、空状态、错误提示、操作确认、结果反馈没有形成统一组件规范。
- 旧平台后台中的机构配置、设备档案、权限菜单、人工修正、面单打印、合并发货、人员绩效等能力尚未形成明确页面规划。

结论：当前前端可以用于验证接口，但不能作为最终后台形态继续堆功能。2026-07-18 已先完成应用壳和现有业务页拆分，后续继续补旧平台缺口能力。

## 2. 重构目标

`admin-web` 的目标不是照搬 JSP 页面样式，而是完整替代旧平台后台的核心操作能力。

第一目标：

- 让药房/运营人员能按真实业务流程处理订单、处方、审核、调剂、复核、煎煮、物流和异常补偿。
- 让运维人员能按订单号、处方号、物流单号、回调记录追踪全链路问题。
- 让后台页面结构能继续承载后续机构配置、设备档案、权限菜单和报表。

非目标：

- 不做营销站、产品官网或大屏驾驶舱。
- 不追求旧 JSP 的像素级还原。
- 不把所有后端调试接口暴露成主菜单。
- 不在前端保存机构密钥、生产密码、私钥或固定登录凭据。

## 3. 方案取舍

### 方案 A：继续在 `App.vue` 上补页面

优点是改动小，短期能继续联调。缺点是单文件会继续膨胀，页面之间无法形成稳定导航和复用组件，后续旧平台功能越补越难维护。

不采用。

### 方案 B：先做后台应用骨架，再迁移现有页面

先建立布局、路由、菜单、业务模块目录、通用组件和 API 边界，再把现有功能按模块迁移进去。这样能保留已接通的接口能力，同时把页面组织方式调整为旧平台后台的业务视角。

采用此方案。

### 方案 C：引入完整后台模板后重写

可以快速得到菜单、权限、表格和主题，但模板通常会带大量无关代码和约定，容易压过当前业务边界，也会增加后续排查成本。

暂不采用。后续如果页面数量和表单复杂度继续上升，可单独评估 Element Plus / Naive UI 等组件库，但不先引入整套后台模板。

## 5. 核心页面设计

### 5.1 工作台

目标：替代当前打开页面后直接进入审核任务的形态，给后台用户一个可扫描的业务入口。

内容：

- 今日待审核、待调剂、待复核、待打包、回调失败、集成失败。
- 最近订单列表。
- 核心健康概览，复用 `ops-service` 的 `health-overview`。
- 快捷入口：推单排查、订单查询、回调失败、集成失败。

验收：用户不需要知道服务名，也能判断当前要优先处理什么。

### 5.2 订单中心

目标：旧后台订单列表和订单详情的替代入口。

列表能力：

- 按订单号、外部订单号、处方号、手机号、机构、订单状态、时间范围筛选。
- 展示订单号、机构、患者脱敏信息、状态、处方数、创建时间、更新时间、最新履约节点。
- 支持进入订单详情。

详情能力：

- 基础信息、处方列表、状态日志、履约时间线、调剂记录、煎煮任务、物流单、回调记录。
- 人工操作入口预留：取消订单、修改地址、备注、人工走流程。未接后端前不展示可点击假按钮。

当前已有 `GET /api/admin/orders/{orderNo}` 和 `/progress`，第一轮先做单号查询详情；列表 API 后续由 `order-service` 补齐后接入。

### 5.3 处方中心

目标：让处方从订单详情里的附属信息变成独立查询和处理对象。

列表能力：

- 按处方号、订单号、机构、处方状态、煎煮状态筛选。
- 展示处方号、订单号、状态、明细数量、审核/调剂/复核状态、煎煮状态。

详情能力：

- 处方基础信息、明细、审核结果、调剂记录、复核记录、煎煮任务。
- 后续补处方修改、处方明细修改、重打、附件查看。

当前后端 `prescription-service` 仍是骨架，第一轮前端只做详情区域和从订单详情跳转的结构预留，不做假数据页面。

### 5.4 药房作业

目标：替代旧后台审核、调剂、复核三个高频工作流。

通用规则：

- 三类任务使用同一套任务列表组件。
- 列表展示订单号、外部订单号、任务状态、订单状态、校验结果、创建时间、更新时间。
- 处理动作必须有确认、操作人、备注。
- 操作成功后刷新当前列表和相关订单详情缓存。

审核任务：

- 支持通过、拒绝。
- 展示校验状态和校验说明。

调剂任务：

- 支持完成调剂。
- 展示或预留打印状态、调剂人、调剂备注、处方信息。

复核任务：

- 支持完成复核。
- 展示或预留复核附件、水桶绑定、复核说明。

### 5.5 煎煮作业

目标：把当前“煎煮模拟”改成业务页，而不是测试工具页。

页面拆分：

- 待煎煮处方：查询复核完成可操作处方，支持绑定设备。
- 煎煮任务：展示任务状态、设备、水桶、开始/完成/取消/终止操作。
- 作业明细：展示事件、温度、加水、异常、拒绝记录。

说明：

- PDA/MES 模拟操作可以保留，但放在“模拟操作区”或开发环境入口，不作为生产主按钮文案。
- 旧 PDA/MES 协议兼容由后端 adapter 负责，前端只展示归一后的业务状态。

### 5.6 物流发货

目标：替代旧后台待发货、面单、打包、自提、物流轨迹和回调补偿入口。

页面拆分：

- 待打包订单：展示 `DECOCTED` 待处理订单，支持打包。
- 物流单：展示物流公司、运单号、状态、包裹数、重量、发货/签收操作。
- 物流轨迹：按物流单展示轨迹。
- 回调补偿：查看回调记录、失败原因、重放、标记成功/失败、派发到期回调。

后续能力：

- 面单查询/打印。
- 物流费用查询。
- 合并发货。
- 自提完成。
- 批量签收。

### 5.7 门户与集成

目标：保留当前医院查单、地址补录、集成消息和重试任务能力，但从“接口模拟”改成运营处理入口。

页面：

- 医院查单：订单号/外部单号 + 手机号查询。
- 地址补录：提交补录申请，展示申请状态。
- 集成消息：按来源、状态、业务键查询。
- 重试任务：派发到期任务，查看失败响应和下次重试时间。

社康完整拉单暂缓，不在当前前端第一轮实现。

### 5.8 报表统计

目标：保留当前业务概览和 CSV 导出，后续扩展人员绩效、药品统计和订单导出。

第一轮：

- 核心指标。
- 订单状态分布。
- 回调状态分布。
- 订单趋势。
- CSV 导出。

后续：

- 订单列表导出。
- 调剂/复核/煎煮人员绩效。
- 药品统计。

### 5.9 运维审计

目标：把排错能力集中到运维审计，而不是散在业务页面。

页面：

- 健康概览。
- Outbox。
- 消费日志。
- 订单校验。
- 访问日志。
- 回调失败。
- 集成失败。

规则：

- 默认展示业务可读字段，如订单号、处方号、物流单号、来源系统、失败原因。
- 原始 payload 和技术字段放到详情抽屉，不作为主表格第一视觉。
- 运维页面只读，补偿动作仍放到对应业务页或明确的补偿按钮中。

拆分原则：

- `App.vue` 只保留应用壳、全局通知和路由出口。

- `features/<module>` 内放对应页面、局部组件和模块 composable。
- 通用表格、筛选、详情抽屉、确认弹窗沉到 `components`。
- 状态中文名、状态颜色、时间格式、金额格式放到 `domain` 或 `formatters`，页面不重复硬编码。

## 7. API 接入边界

当前 API 文件可以保留，但页面层不能直接拼接技术路径或散落错误处理。

规则：

- 每个业务模块只通过自己的 API facade 调后端，例如订单页面只依赖 `order.ts` 和必要的 progress 查询。
- API 返回值在页面展示前转换为前端 view model，避免表格里直接展示后端枚举和原始字段。
- 请求错误统一进入 `useAsyncState` 或模块级错误处理，避免每个按钮重复写 try/catch。
- 后端未实现的能力不得在前端做假按钮；可以在方案和 TODO 清单中列出。

需要后端后续补齐的接口：

- 订单列表分页查询。
- 处方列表和处方详情查询。
- 机构、应用密钥、回调地址、白名单配置接口。
- 设备档案和设备占用查询接口。
- 人工修正接口：取消、初始化、人工走流程、手工签收、备注。
- 面单查询/打印、物流费用、合并发货。
- 用户、角色、菜单权限接口。

## 8. 状态与交互规范

表格：

- 固定筛选区、结果区、分页区。
- 高风险动作使用确认弹窗。
- 操作列只放当前状态允许的动作，不显示不可用按钮堆叠。
- 长 payload 使用详情抽屉，不直接撑开表格。

详情：

- 订单、处方、物流单、回调记录统一使用右侧抽屉或详情页。
- 详情页必须显示时间线或状态日志，便于追踪。
- 从异常记录能跳转到相关订单、物流单或集成消息。

状态：

- 主表格展示中文业务状态。
- 技术枚举可放在 tooltip、详情抽屉或运维页。
- 状态颜色保持一致：待处理、处理中、成功、失败、终态、取消。

反馈：

- 查询无数据要说明筛选条件下暂无记录。
- 操作成功提示业务结果。
- 操作失败提示后端业务错误，不只显示网络失败。

## 9. 分阶段落地

### 阶段一：前端骨架和现有能力迁移

范围：

- 建立 `app/router.ts`、`app/menu.ts`、`AppLayout`。
- 拆出工作台、订单中心、药房作业、报表统计、运维审计。
- 把现有审核、调剂、复核、订单查询、报表、运维能力从 `App.vue` 迁移到模块页面。
- 保持现有 API 不变，确保 `pnpm build` 通过。

验收：

- `App.vue` 不再承载具体业务表格和操作逻辑。
- 页面入口按业务菜单组织。
- 当前已接通的订单查询、审核、调剂、复核、报表导出、健康概览可用。

### 阶段二：煎煮、物流、门户集成迁移

范围：

- 拆出煎煮作业、物流发货、门户与集成模块。
- 把当前煎煮模拟、物流回调、医院查单、地址补录、集成适配迁移成业务页面。
- 增加详情抽屉和通用表格能力。

验收：

- 煎煮、物流、回调、集成重试不再集中在 `App.vue`。
- 运维异常可以跳转到相关业务记录。
- 页面文案从“模拟/适配/排错”调整为业务人员能理解的动作。

当前状态：

- 已完成页面拆分：`features/decoction/DecoctionWorkspace.vue`、`features/logistics/LogisticsFulfillment.vue`、`features/portal/PortalLookup.vue`、`features/integration/IntegrationConsole.vue`。
- `App.vue` 已不再承载煎煮、物流、门户和集成页面的业务表格、筛选和操作逻辑。

### 阶段三：补齐旧平台后台核心缺口

范围：

- 订单列表、处方列表、处方详情。
- 人工修正入口。
- 面单、合并发货、自提、批量签收。
- 机构配置、回调配置、设备档案。
- 用户、角色、菜单权限。

验收：

- `docs/02_重构规格/新旧功能映射.md` 中后台必做项有对应页面或明确后端阻塞说明。
- 旧平台后台核心操作可以在新 `admin-web` 中完成。

### 阶段四：联调验收和体验收口

范围：

- 跑通推单、审核、调剂、复核、煎煮、打包、发货、签收、回调、报表、运维排错。
- 检查响应式布局、文字溢出、按钮拥挤、表格横向滚动。
- 补充操作日志和错误提示。

验收：

- `pnpm build` 通过。
- 核心页面无空壳入口。
- 关键工作流均有真实接口验证记录。

## 10. 当前优先级

下一步前端重构优先级：

1. 补旧平台后台缺失能力对应页面：订单列表、处方中心、机构配置、设备档案、权限菜单和人工修正入口。
2. 对现有 feature 页面继续收口为通用表格、筛选、详情抽屉和确认动作组件。
3. 后端未实现的能力同步记录阻塞，不在前端做假数据页面。
4. 做前后端真实联调和验收。

当前不优先：

- 社康完整拉单后台配置。
- AI 页面。
- 大屏驾驶舱。
- 复杂主题系统。
- 纯视觉重做。
