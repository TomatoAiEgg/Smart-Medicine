## 5. 当前后端模块

```text
backend
├── zhyf-common
├── gateway
├── auth-institution
├── order-service
├── prescription-service
├── message-service
├── workflow-service
├── ops-service
├── decoction-service
├── logistics-service
└── callback-service
```

第一阶段实际承载：

- `gateway`：机构推单入口、签名/IP 校验、访问日志、转发到 `order-service`。
- `auth-institution`：机构应用和 IP 白名单内部查询。
- `order-service`：机构推单、订单/处方/明细落库、状态日志、Outbox、订单查询、内部订单状态推进接口。
- `message-service`：Outbox 发布、RocketMQ 生产者、RocketMQ 消费入口、消费幂等、`ORDER_CREATED` 校验记录。
- `workflow-service`：消费校验通过的 `ORDER_CREATED` 创建待审核任务、审核任务查询、审核通过/拒绝、调用 `order-service` 推进订单审核状态。
- `workflow-service`：消费校验通过的 `ORDER_CREATED` 创建待审核任务、审核任务查询、审核通过/拒绝、审核通过后创建复核任务、复核完成时调用 `order-service` 推进订单到 `RECHECKED`。
- `prescription-service`：当前只有模块骨架，默认本地端口 `18084`，处方表第一阶段暂由 `order-service` 承载。
- `ops-service`：默认本地端口 `18086`，已提供只读排错查询接口，用于联调查看 Outbox、消费日志、订单校验记录、网关访问日志，以及物流/回调失败原因聚合；`admin-web` 已接入“运维排错”页。
- `decoction-service`：默认本地端口 `18087`，已提供 PDA/MES 模拟器第一版，并已通过真实本地联调验证从复核完成处方绑定设备、开始煎煮、完成煎煮并推进订单状态；当前已补加水完成、温度、异常事件记录、取消/终止、设备释放规则、非法状态拒绝事件记录、设备作业明细、真实设备 adapter 抽象和旧 PDA/MES 协议兼容入口。
- `logistics-service`：默认本地端口 `18088`，已提供物流/自提最小链路：待打包订单查询、打包、发货、签收、顺丰/EMS 轨迹回调兼容入口、物流轨迹查询，并通过 `order-service` 内部接口推进订单到 `PACKED/SHIPPED/IN_TRANSIT/SIGNED`。
- `callback-service`：默认本地端口 `18089`，已提供机构回调记录补偿模型：创建 `callback_record`、真实 HTTP 回调发送、到期回调手动派发、可配置定时派发、失败重试、`DEAD` 终态、人工标记成功/失败和重放；定时派发默认关闭，避免开发环境误触发外部接口。

## 6. 当前已完成链路

当前最小链路已经推进到：

```text
机构推单
 -> gateway 签名/IP/访问日志
 -> order-service 创建订单
 -> order-service 创建处方和处方明细
 -> order_status_log 写 CREATED
 -> event_outbox 写 ORDER_CREATED
 -> message-service 发布 RocketMQ
 -> message-service 消费 ORDER_CREATED
 -> message_consume_log 幂等记录
 -> order_validation_record 写 PASSED/REJECTED
 -> workflow-service 查询校验记录并创建 workflow_task: ORDER_REVIEW/PENDING
 -> workflow-service 查询待审核任务
 -> workflow-service 审核通过/拒绝
 -> order-service 内部接口推进 order_main.status=AUDIT_PASSED/AUDIT_FAILED
 -> order_status_log 写审核状态变化
 -> 审核通过后 workflow-service 创建 workflow_task: PRESCRIPTION_RECHECK/PENDING
 -> admin-web 查询复核任务
 -> workflow-service 完成复核
 -> order-service 内部接口推进 order_main.status=RECHECKED
 -> order_status_log 写复核状态变化
 -> admin-web 煎煮模拟页查询可操作处方和设备
 -> decoction-service 创建 decoction_task: BOUND
 -> MES/PDA 模拟开始煎煮
 -> order-service 内部接口推进 order_main.status=DECOCTING
 -> MES/PDA 模拟完成煎煮
 -> order-service 内部接口推进 order_main.status=DECOCTED
 -> order_status_log 写煎煮状态变化
 -> MES 模拟记录加水完成/温度/异常事件到 decoction_task_event
 -> BOUND 任务可取消为 CANCELLED 并释放设备，订单保持 RECHECKED 可重新绑定
 -> DECOCTING 任务可终止为 TERMINATED，并推进订单 DECOCTING -> CANCELLED
 -> 非法状态/乱序上报会被拒绝，并写入 decoction_task_event: REJECTED
 -> 成功和拒绝的设备动作写入 decoction_device_work_record 结构化作业明细
 -> admin-web 运维排错页可查询 Outbox、消费日志、订单校验记录和网关访问日志
 -> 旧 PDA/MES 协议入口复用 decoction-service adapter 映射到同一套煎煮业务服务
 -> admin-web 物流回调页查询 DECOCTED 待打包订单
 -> logistics-service 创建 shipment: PACKED，并推进订单 DECOCTED -> PACKED
 -> logistics-service 发货/签收或接收顺丰/EMS 轨迹回调，写 shipment_trace
 -> order-service 内部接口推进订单 PACKED -> SHIPPED/IN_TRANSIT/SIGNED
 -> callback-service 创建 callback_record，支持真实 HTTP 派发、失败重试、DEAD 终态和人工成功/失败/重放
 -> admin-web 物流回调页可查询物流单、轨迹和回调补偿记录
```
