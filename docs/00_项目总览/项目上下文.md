## 2. 旧系统背景

旧项目主要位于：

```text
D:\workproject\chengyou
```

重点分析过的系统：

- `1\zhyfsupport`
- `1\zhyf-api`
- `sc-zhyf-api`
- `sc-zhyf-communityhospital`
- `sc-address-input`
- `hospital-query`

核心判断：

旧系统不是严格意义上的微服务，而是多个应用围绕同一套智慧药房数据库协作的集中式业务系统。

主要问题：

- 多项目共享同一个 MySQL 核心库。
- 业务边界不清晰。
- 配置、密钥、Redis、MQ、数据库连接散落在配置文件中。
- PDA、煎煮、物流、回调状态分散。
- ActiveMQ 只是消息通道，最终状态仍然落共享库。
- JSP 后台维护成本高。
- 测试薄弱，压测和监控缺失。

## 9. PDA 和 MES 处理方式

当前没有真实 PDA 和煎煮设备。

处理策略：

- PDA 先做模拟器。
- MES 先做模拟器。
- decoction-service 中抽象 `MesAdapter`。
- 第一阶段使用 `MockMesAdapter`。
- 后续真实 MES 接入时实现 `HttpMesAdapter`。

PDA 模拟重点：

- 登录
- 获取待作业处方
- 扫码绑定
- 加水开始
- 加水完成
- 煎煮开始
- 煎煮完成
- 重复扫码
- 并发扫码
- 网络超时后重试

MES 模拟重点：

- 接收煎煮任务
- 上报开始煎煮
- 上报完成煎煮
- 上报异常
- 乱序上报
- 重复上报

## 10. RocketMQ 事件方向

核心事件：

- `ORDER_CREATED`
- `PRESCRIPTION_CREATED`
- `PRESCRIPTION_REVIEWED`
- `PDA_PRESCRIPTION_BOUND`
- `DECOCTION_WATERED`
- `DECOCTION_STARTED`
- `DECOCTION_FINISHED`
- `ORDER_PACKED`
- `LOGISTICS_SHIPPED`
- `LOGISTICS_IN_TRANSIT`
- `LOGISTICS_SIGNED`
- `CALLBACK_REQUESTED`
- `CALLBACK_SUCCEEDED`
- `CALLBACK_FAILED`

事件设计原则：

- 业务写库和写 `event_outbox` 在同一事务中完成。
- 后台任务扫描 `event_outbox` 发送 RocketMQ。
- 消费端使用 `message_consume_log` 做幂等。
- 回调失败由 callback-service 管理重试节奏。
- 超过最大重试次数进入死信，后台人工补偿。

## 12. 状态机方向

订单状态：

```text
CREATED
 -> PRESCRIPTION_CREATED
 -> REVIEWED
 -> DISPENSED
 -> RECHECKED
 -> DECOCTING
 -> DECOCTED
 -> PACKED
 -> SHIPPED
 -> IN_TRANSIT
 -> SIGNED
```

处方状态：

```text
PENDING_REVIEW
 -> REVIEW_PASSED
 -> DISPENSING
 -> DISPENSED
 -> RECHECKED
 -> FINISHED
```

煎煮状态：

```text
WAIT_BIND
 -> BOUND
 -> WATERING
 -> WATERED
 -> DECOCTING
 -> DECOCTED
 -> PACKED
```

要求：

- 状态字段只允许状态机组件修改。
- 状态变更必须写日志。
- 非法状态跳转必须拒绝。
- 并发操作必须通过版本号或唯一约束控制。

## 13. 压测目标

压测不压空接口，要压真实业务链路。

核心压测场景：

- 机构批量推单
- PDA 并发扫码绑定
- MES 并发状态上报
- 物流轨迹批量回调
- 医院/社康回调失败重试
- 后台订单分页查询
- 后台物流轨迹查询

初始性能目标：

- 推单接口 P95 小于 500ms。
- PDA 操作接口 P95 小于 300ms。
- MES 上报接口 P95 小于 300ms。
- 回调接收接口 P95 小于 300ms。
- 后台常用查询 P95 小于 800ms。
- RocketMQ 不出现长时间堆积。
