# 本地开发环境

本文档记录本地开发所需工具、环境变量、服务启动顺序和联调检查项。

## 1. 基础工具

| 工具 | 建议版本 | 用途 |
| --- | --- | --- |
| JDK | 21 | 后端服务开发和编译 |
| Maven | 3.9.x | 后端多模块构建 |
| Node.js | 20 LTS 或 22 LTS | 前端管理台开发 |
| pnpm | 9.x 或更高 | 前端依赖管理 |
| Git | 2.x | 版本管理 |
| IntelliJ IDEA | 2024.x 或更高 | Java 开发 |
| VS Code | 最新稳定版 | 前端开发 |

IDEA 当前应导入 `backend/pom.xml` 作为 Maven 项目，Project SDK 使用 JDK 21。

## 2. 当前中间件策略

当前开发默认连接老笔记本 Ubuntu 上的项目专用中间件：

- PostgreSQL：`100.115.165.51:15432`
- Redis：`100.115.165.51:16379`
- RocketMQ NameServer：`100.115.165.51:9876`
- RocketMQ Broker：`100.115.165.51:10911`
- RocketMQ Dashboard：`http://100.115.165.51:18181`
- Grafana：`http://100.115.165.51:13000`
- Loki：`http://100.115.165.51:13100`
- Prometheus：`http://100.115.165.51:19090`

本地 Redis / RocketMQ / PostgreSQL / Docker Desktop 默认不启动，不作为当前开发依赖。`infra/docker-compose.local-optional.yml` 只作为离线备用方案，必须明确决定脱离远端中间件后才能手动使用。

应用服务默认在本地 IDEA / Maven 启动，不默认上传 jar 到服务器运行。

## 3. 环境变量建议

敏感值不要写入代码仓库。建议在本机 shell、IDEA Run Configuration 或本地未提交配置文件中维护。

```text
ZHYF_PROFILE=dev
ZHYF_DB_HOST=100.115.165.51
ZHYF_DB_PORT=15432
ZHYF_DB_NAME=zhyf_saas
ZHYF_DB_USERNAME=<username>
ZHYF_DB_PASSWORD=<password>
ZHYF_REDIS_HOST=100.115.165.51
ZHYF_REDIS_PORT=16379
ZHYF_REDIS_PASSWORD=<password>
ZHYF_ROCKETMQ_NAMESRV=100.115.165.51:9876
ZHYF_CALLBACK_DISPATCH_ENABLED=false
```

Windows 命令行如需临时使用 JDK 21：

```text
$env:JAVA_HOME='D:\Javasource\javaTools\jdk-21.0.10'
$env:Path="$env:JAVA_HOME\bin;$env:Path"
```

## 4. SSH 隧道方式

本机通过 Tailscale 直连失败时，可以开启 SSH 隧道：

```text
scripts/start-cloud-middleware-tunnel.ps1
```

隧道打开后，本地应用可以改连：

```text
PostgreSQL: 127.0.0.1:15432
Redis: 127.0.0.1:16379
RocketMQ NameServer: 127.0.0.1:9876
RocketMQ Broker: 127.0.0.1:10911
RocketMQ Dashboard: http://127.0.0.1:18181
```

脚本不会写入密码；私钥文件不应提交。

兼容说明：`scripts/start-cloud-middleware-tunnel.ps1` 仍保留，但只是转调当前脚本，后续新命令使用 `scripts/start-middleware-tunnel.ps1`。

## 5. 后端启动顺序

第一版推荐顺序：

```text
auth-institution
order-service
message-service
workflow-service
gateway
prescription-service
ops-service
decoction-service
callback-service
logistics-service
portal-service
report-service
```

开发早期可以先启动：

```text
auth-institution
order-service
message-service
gateway
```

要求：

- 启动前确认数据库可连接，Flyway 能执行到最新版本。
- 每个服务使用独立端口。
- 日志可以打印 profile、数据库地址、Redis 地址、RocketMQ NameServer，但不能打印密码。
- `message-service` 默认使用 logging 发布器；需要真实发布 RocketMQ 时再设置 `ZHYF_MESSAGE_PUBLISHER=rocketmq`。

## 6. 本地启动脚本

```text
scripts/run-auth-institution-local.ps1
scripts/run-order-service-local.ps1
scripts/run-message-service-local.ps1
scripts/run-gateway-local.ps1
```

脚本不保存密码，数据库和 Redis 密码通过当前 PowerShell 环境变量或运行时输入提供。

## 7. 前端启动

```text
cd frontend/admin-web
pnpm install
pnpm dev
```

默认访问地址：

```text
http://127.0.0.1:5174/
```

当前 `admin-web` 提供订单查询、workflow 审核/复核任务、煎煮模拟、物流回调、门户查单、报表统计和运维排错。Vite 开发代理默认转发：

- `/order-api` -> `http://127.0.0.1:18082`
- `/workflow-api` -> `http://127.0.0.1:18085`
- `/decoction-api` -> `http://127.0.0.1:18087`
- `/ops-api` -> `http://127.0.0.1:18086`
- `/logistics-api` -> `http://127.0.0.1:18088`
- `/callback-api` -> `http://127.0.0.1:18089`
- `/portal-api` -> `http://127.0.0.1:18090`
- `/report-api` -> `http://127.0.0.1:18091`

使用前按页面需要启动对应服务：订单查询需要 `order-service`，审核/复核需要 `workflow-service`，煎煮模拟需要 `decoction-service`，物流回调需要 `logistics-service` 和 `callback-service`，门户查单需要 `portal-service`，报表统计需要 `report-service`，运维排错需要 `ops-service`。运维排错中的“回调失败”会关联 `callback_record`、`shipment` 和最新 `shipment_trace`，用于定位物流回调失败原因。如果页面返回后端错误，优先看对应服务控制台。

前端环境变量只保存 API 地址，不保存机构密钥和管理端账号密码。

`callback-service` 定时派发外部 HTTP 回调默认关闭。需要自动派发时设置：

```text
ZHYF_CALLBACK_DISPATCH_ENABLED=true
ZHYF_CALLBACK_DISPATCH_FIXED_DELAY_MS=5000
ZHYF_CALLBACK_MAX_RETRIES=5
```

开发阶段也可以在 `admin-web` 的“物流回调 / 回调记录”页点击“派发到期回调”，手动触发当前到期的 `PENDING/FAILED` 记录。

## 8. 联调检查清单

- PostgreSQL 能连接项目库 `zhyf_saas`。
- Redis `PING` 正常。
- RocketMQ Dashboard 可打开，Broker 在线。
- Grafana 健康检查正常，Loki 能查询到 Docker 日志标签，Prometheus 正常 ready。
- `gateway` 健康检查正常。
- 机构签名校验能通过正常请求并拒绝错误签名。
- 创建订单后能写入订单、处方、处方明细和 `event_outbox`。
- Outbox 能发布 RocketMQ，消费日志能记录幂等结果。
- 煎煮完成后，`logistics-service` 能查询 `DECOCTED` 待打包订单并创建 `shipment`。
- 物流发货、签收或顺丰/EMS 轨迹回调能写入 `shipment_trace` 并推进订单状态。
- `callback-service` 能写入 `callback_record`，管理端能人工标记成功、失败、重放或派发到期回调。
- 外部回调发送成功后 `callback_record.status=SUCCESS`；失败未达上限时写 `FAILED` 和 `next_retry_at`；达到上限时写 `DEAD`。
- `ops-service` 的“回调失败”查询能看到 `FAILED/DEAD` 回调、物流单、订单号、失败响应和最新轨迹。
- `portal-service` 查单必须使用订单号或外部单号加手机号；地址补录申请写入 `portal_address_supplement.PENDING`，不直接修改 `order_main` 地址字段。
- `report-service` 报表统计为只读聚合，能返回订单、处方、物流、回调、待处理地址补录、订单状态分布、回调状态分布和近期订单趋势。
